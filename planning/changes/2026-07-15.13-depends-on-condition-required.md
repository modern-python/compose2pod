# Design: depends_on long-form condition becomes required

## Summary

`graph.depends_on`'s long-form branch defaulted a missing `condition` to
`service_started` (`spec.get("condition", "service_started")`). Measured
against live `docker compose config` v5.1.2, Docker refuses that same
document ("missing property 'condition'") while still silently defaulting
the *short* (list) form. This was flagged, not fixed, in `2026-07-15.12`'s
Non-goals as the named revisit trigger from
`planning/decisions/2026-07-14-docker-rejection-parity.md` ("Docker's own
validation changes such that a construct compose2pod correctly accepts
starts being refused"). The maintainer has ruled: match Docker — the long
form requires an explicit `condition`; the short form keeps its default.

## Motivation

Measured, same YAML both oracles:

- `depends_on: {db: {}}` — Docker REJECT; compose2pod ACCEPTed (defaulted).
- `depends_on: {db: {restart: true}}` (long form, no `condition`) — Docker
  REJECT; compose2pod ACCEPTed.
- `depends_on: {db: {condition: service_started}}` — Docker ACCEPT (unchanged).
- `depends_on: [db]` (short form) — Docker ACCEPT, defaults to
  `service_started` (unchanged; Docker's own default, not ours).

This was the one remaining false green against the hard rule
`accepted(compose2pod) ⊆ accepted(docker)`.

## Design

**`compose2pod/graph.py`, `_depends_on_entry_condition`:** after the existing
unknown-key check, a long-form entry with no `condition` key raises
`UnsupportedComposeError` ("missing required key 'condition'") before the
type/membership checks run. The short-form branch of `depends_on` itself
(`dict.fromkeys(deps, "service_started")`) is untouched.

**`compose2pod/extends.py`, `_as_mapping`'s depends_on list->mapping
normalizer:** this runs ahead of `validate()` (`extends` resolution is
first) and previously produced `{dep: {}}` per short-form entry when merging
against a base that also defines `depends_on` — a condition-less long-form
entry, which the new gate now refuses. A short-form `depends_on` that merges
through `extends` would therefore have started failing even though the same
document is valid standalone. Fixed by injecting the default explicitly:
`{dep: {"condition": "service_started"}}`. This is the one interaction the
new gate could have broken and was caught by writing the extends-chain
regression test first (see Testing) — confirmed failing (`missing required
key 'condition'`) before the `extends.py` fix, passing after.

`startup_order` (`graph.py`) only reads `depends_on(svc)`'s keys for its
closure walk, never `condition`, so it needed no change; confirmed by the
full `tests/test_graph.py` suite passing unchanged.

## Non-goals

Matching Docker's message text — only a clean `UnsupportedComposeError` is
required, not "missing property 'condition'" verbatim.

## Testing

**Unit** (TDD — written first, confirmed `DID NOT RAISE` against pre-fix
code): `tests/test_graph.py::TestDependsOn` —
`test_long_form_missing_condition_raises`,
`test_long_form_with_other_sub_keys_but_no_condition_still_raises`. Four
existing tests that exercised `restart`/`required` on a condition-less
long-form entry (`test_restart_must_be_a_boolean`,
`test_required_must_be_a_boolean`,
`test_restart_variable_reference_passes_through`,
`test_required_variable_reference_passes_through`) gained an explicit
`condition` key so they keep testing what they name rather than tripping the
new check first.

`tests/test_extends.py` — `test_short_form_depends_on_through_extends_stays_valid`
(new; the extends-normalization regression above, confirmed failing before
the `extends.py` fix), `test_depends_on_list_and_map_forms_merge`'s
assertion and the `HOSTILE` table's `depends_on` entry updated to the new
explicit-condition shape.

`tests/integration/test_depends_on_default.py`: switched its
`depends_on: {"helper": {}}` to the short form `depends_on: ["helper"]` —
the scenario it verifies (service_started's zero-synchronization default) is
still exactly the short form's default, which is unchanged; the long form
just no longer means the same thing. `pytest -m integration
tests/integration/test_depends_on_default.py`: 1 passed (podman on PATH).

`just test-ci`: 1227 passed, 100% line coverage.

**Conformance**: new corpus file
`tests/conformance/corpus/depends_on_long_form_missing_condition.yaml`
(`depends_on: {db: {restart: true}}`, no `condition`). Confirmed failing
(`AssertionError: docker compose config REJECTS this document but
compose2pod ACCEPTS it`) against pre-fix `graph.py`/`extends.py`
(`git stash`), passing after. `just test-conformance`: 784 passed, 4

edges, `env_file`/duration host-state carve-outs); none newly introduced.

`architecture/supported-subset.md`'s `## depends_on` section rewritten:
states the two forms' differing defaults directly (short form defaults,
long form requires `condition`) and documents the `extends.py` normalization
fix.

`just lint-ci` and `just check-planning` both clean.

## Risk

Low and one-directional (narrows acceptance, the direction the parity rule
requires). The only document that stops validating is a long-form
`depends_on` entry with no `condition` — which Docker itself already
refused, so no compose2pod user could have run that document through `docker
compose` successfully either.
