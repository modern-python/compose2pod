# Design: Per-key value grammars for `build`

## Summary

Task 4 (`2026-07-13.10`) checked `build`'s mapping form against Docker's own
22-key schema, but only the key *names* — `build: {context: 3}` and
`no_cache: 5` were still accepted here, both documents `docker compose
config` refuses. `_DOCKER_BUILD_KEYS` is now `dict[str, Callable[[str, str,
Any], None]]`, mirroring `IGNORED_SERVICE_KEYS`'s existing set→dict shape:
every known key's value is checked against its own measured grammar before
`_validate_build` returns. compose2pod still never *emits* any of it
(`image_for` always substitutes `--image`), so nothing here is enumerated
because compose2pod needs it — only because a malformed value is a document
Docker will not run.

## Motivation

`planning/decisions/2026-07-14-docker-rejection-parity.md`'s hard rule —
`accepted(compose2pod) ⊆ accepted(docker)`, no exceptions — bound `build`'s
key *names* already but not one value inside it. The generated conformance
matrix (`tests/conformance/test_matrix.py`) probes one *service* key at a
time and can never reach a nested position like `build.context`, so this
gap could not close itself the way the top-level scalar grammars did.
Measured by hand against live `docker compose config` v5.1.2 (same YAML text
fed to both oracles: `compose2pod.cli._read_compose` + `compose2pod.parsing.
validate` vs `docker compose config -f`), all 22 keys carry a real grammar,
grouped into six clusters (string, size, bool, list-of-strings, list-or-map,
nested) — see `_DOCKER_BUILD_KEYS` in `compose2pod/parsing.py` for the full
per-key map.

## Design

**String cluster** (`context`, `dockerfile`, `dockerfile_inline`, `target`,
`network`, `isolation`) reuses `values.validate_string` directly — no new
code, since a `${VAR}` reference is already a valid string and needs no
special-casing here.

**Size cluster** (`shm_size`) reuses `values.validate_size`, but with
`allow_fractional=False` — measured divergence from the *top-level*
`shm_size` service key (which keeps the default `allow_fractional=True`):
build's own `shm_size` refuses *any* fractional float (`0.5`), whole or not,
while the top-level key accepts one. Same identifier, different Go field,
different grammar — re-measured rather than assumed shared.

**Bool cluster** (`no_cache`, `pull`, `privileged`) is a new
`_validate_build_bool`: strict `isinstance(value, bool)`, with an explicit
`values.has_variable` short-circuit ahead of it. This is the one cluster
where the ${VAR} carve-out is load-bearing rather than automatic: measured,
`no_cache: ${MYVAR}` is accepted when `MYVAR=true` and refused when unset or
non-boolean — genuinely host-state-dependent, unlike (say) `cache_from:
${MYVAR}`, which Docker refuses unconditionally regardless of what `MYVAR`
holds (a scalar can never satisfy an array-typed field, so that refusal
binds regardless of the reading shell). `no_cache: "true"` (a literal quoted
string, not a variable reference) stays refused — a deferred over-rejection,
consistent with the top-level six boolean keys' existing limitation note in
`planning/deferred.md`, now extended to cover this cluster too.

**List-of-strings cluster** (`cache_from`, `cache_to`, `tags`, `platforms`,
`entitlements`) reuses `parsing._validate_string_list`, already defined for
`profiles`. **`ssh` was NOT added here** — see Non-goals.

**List-or-map cluster.** Three different grammars, not one:

- `args`/`labels`/`ssh` → new `_validate_build_map`, wrapping
  `keys.validate_map` (a list of `'KEY[=value]'` strings — bare `'KEY'`
  means a null value — or a mapping with scalar-or-null values). `ssh` was
  measured to share this *exact* grammar with `args`/`labels`, not the
  list-of-strings shape its docs suggest at a glance (`docker compose
  config` accepts `build.ssh: {default: /path}` too). The wrapper adds a
  `require_string_keys` check `validate_map` itself does not make: the
  top-level `labels`/`environment` get that check for free from
  `parsing._sweep_service`'s deep sweep, but `_sweep_service` explicitly
  skips `build`'s contents, so build's own mapping-shaped values need their
  own check. Measured: Docker refuses a non-string key here too
  (`non-string key in services.app.build.args: true`).
- `additional_contexts` → new `_validate_build_additional_contexts`:
  stricter than args/labels/ssh in two measured ways. A list entry must
  contain `'='` (no bare-`'KEY'` null-value form). A map value must be a
  plain string — a number/bool/null passes Docker's own JSON schema (the
  same scalar-or-null union args/labels/ssh accept) but then crashes
  Docker's own context-path resolver downstream (`compose-go` v2.10.2:
  `interface conversion: interface {} is int, not string`, a real bug in
  the dependency — but still exit code 2, "docker rejects" by the harness's
  own definition, so accepting it here would be a document already broken
  upstream).
- `extra_hosts` → new `_validate_build_extra_hosts`: a list entry needs
  either separator (`'host=ip'` or `'host:ip'`, both measured accepted). A
  map value may be a single address string, *or a list of address strings*
  (one host, several IPs) — including an *empty* list, measured accepted
  (an over-rejection trap: it reads like it should be refused, but Docker
  takes it) — but not a number/bool/null.

**Nested cluster.** `ulimits` reuses `keys.validate_ulimits`, renamed from
the module-private `_validate_ulimits` (mirrors `validate_map`'s existing
public-helper pattern): measured identical to the top-level `ulimits`
service key's grammar. The rename adds `require_string_keys` calls at both
nesting levels (outer limit names, inner `soft`/`hard`) — redundant
defense-in-depth for the top-level key (already swept), load-bearing for
`build.ulimits` (which bypasses the sweep, same reasoning as the list-or-map
wrapper above).

`secrets` is new (`_validate_build_secrets`): a list of either a bare
secret-name string, or a `{source, target}` mapping (`target` optional,
both must be strings when present, no other key allowed). See Non-goals for
what this deliberately does not check.

**`_validate_build`** now iterates `_DOCKER_BUILD_KEYS.items()` after the
existing unknown-key check, calling each present key's validator — the same
pattern `_validate_service`'s `SERVICE_KEYS` loop already uses.

## Non-goals

**`build.secrets`'s cross-reference to the top-level `secrets:` store is not
checked.** Measured: `build: {secrets: [{source: doesnotexist}]}` is refused
by Docker (`refers to undefined build secret`) even though it is shape-valid
by every rule above. Closing this needs `_validate_build` to see the whole
document, not one service — a structural change beyond a value-*type*
check, and out of scope here (flagged per the task brief's instruction to
surface, not silently invent-or-skip, a residual gap). A handful of other
narrow shape edges in the `secrets` mapping form (an entry present but with
neither `source` nor `target` set) fall out of the same non-goal.

**Non-string keys are still not checked in the two regions genuinely never
read**: `build`'s own `x-`-prefixed extension keys (arbitrary payload, by
the same convention as everywhere else `x-` appears), and the ignored
top-level `networks`/`volumes` blocks — both unaffected by this change.

## Testing

**Unit** (`tests/test_parsing.py`, `tests/test_keys.py`): one parametrized
test per cluster (string/shm_size/bool/list-of-strings/args-labels-ssh) plus
one dedicated test each for `additional_contexts`, `extra_hosts`, `secrets`,
`ulimits`, and the build-bypasses-the-sweep non-string-key case
(`test_build_args_non_string_key_rejected`). Confirmed red (`DID NOT RAISE`,
39 failures) before implementing, green after. `keys.validate_ulimits`'s
rename updated `tests/test_keys.py`'s import and one identity-comparison
list. `just test-ci`: 1076 passed, 100% line coverage.

**Conformance** (`tests/conformance/corpus/`): 13 new hand-authored
documents, one per representative false green — a string key with an int
(`build_context_int`), a list-of-strings key with a bare string
(`build_cache_from_string`) and a non-string element
(`build_tags_non_string_element`), a bool key with an int
(`build_no_cache_int`), `shm_size` with a non-size string and a fractional
float (`build_shm_size_bad_string`, `build_shm_size_fractional_float`), the
list-or-map cluster's bare-string form (`build_args_bare_string`) and its
non-string-key trap (`build_args_non_string_key`), `ssh`'s corrected list-
or-map classification (`build_ssh_int`), `additional_contexts`'s stricter
bare-key rule (`build_additional_contexts_bare_key`), `extra_hosts`'s
map-value type (`build_extra_hosts_map_int_value`), `secrets`'s list
requirement (`build_secrets_bare_string`), and `ulimits`'s mapping
requirement (`build_ulimits_list`). Verified each fails (`AssertionError`:
docker rejects, compose2pod accepted) against the pre-fix code via `git
stash` (13/13 failed), and passes after. `just test-conformance`: 762
selected, 758 passed, 4 skipped, 0 failed — 0 hard violations; 12
over-rejections, all pre-existing and catalogued in `planning/deferred.md`,
none touching `build`.

`just lint-ci` and `just check-planning` both clean.

## Risk

Low for the six measured clusters: each narrows what was previously
accepted unconditionally, and the parity rule only forbids the opposite
direction (accepting what Docker refuses). The main risk was assuming a
key's grammar from its name or from a sibling with the same name at a
different nesting level — mitigated by measuring every cluster directly
rather than inferring: `ssh` turned out not to be list-of-strings despite
being grouped with `cache_from`/`tags`/etc. in the task brief, `shm_size`
turned out not to share the top-level `shm_size` key's `allow_fractional`
setting, and `extra_hosts`'s empty-list map value turned out to be accepted
despite reading like it should not be. `build.secrets`'s cross-reference gap
(Non-goals) is the one deliberately-left-open piece of the ~38 measured
false greens; it is narrow (a `source` naming an undeclared top-level
secret) and documented above rather than silently absent.
