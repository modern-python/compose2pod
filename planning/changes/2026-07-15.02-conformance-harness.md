# Design: The conformance harness

## Summary

`2026-07-14.10` closed 113 measured divergences from `docker compose config`,
but nothing kept that rule true going forward — five prior changes each
hand-found one divergence, which is what a missing invariant looks like from
the outside. This adds `tests/conformance/`, a CI-only pytest suite (mirroring
`tests/integration/` in shape) that generates a probe matrix from the service-key
registry itself and asserts: **Docker rejects ⇒ compose2pod must reject.**
Where Docker accepts and compose2pod does not, that is reported as an
`over-reject`, not a failure — it is a tracked limitation
(`planning/deferred.md`), not a bug.

## Motivation

A hand-measured table is a claim typed into a file — it cannot catch a
construct nobody thought to enumerate. `tests/conformance/test_matrix.py`
generates its probes from `SERVICE_KEYS | STRUCTURAL_KEYS | IGNORED_SERVICE_KEYS`,
so a key added to any of those registries is probed the moment it lands, and
the rule cannot silently decay as the subset grows.

## Design

**Both oracles read the identical YAML text.** `assert_rule` (`tests/conformance/conftest.py`)
serializes one compose dict with `yaml.safe_dump` and feeds the same text to
`docker compose -f <file> config` (exit code = Docker's verdict; no daemon
needed) and to the real CLI pipeline (`compose2pod.cli._read_compose` →
`resolve_extends` → `validate` → `emit_script` — not `validate()` alone, since
`emit_script` catches what `validate` does not, e.g. an unknown `depends_on`
target). Handing Docker a file and compose2pod a Python dict would let the two
YAML parsers resolve types differently and produce false verdicts in both
directions.

**The generated matrix** (`test_matrix.py`): every key in the registry ×
12 hostile shapes (null, empty str/list/map, bool, int, float, str,
list-of-str, list-of-map, map-str, nested-map) on a single service, asserted
through `assert_rule`.

**The hand-authored corpus** (`test_corpus.py` + `corpus/*.yaml`): six
documents a single-key matrix cannot reach — an `extends` cycle, an unknown
`depends_on` target, a bad `depends_on` condition, a null healthcheck `test`,
an unknown `deploy.resources.limits` key, and a `secrets` entry naming an
undeclared source.

**The `env_file` carve-out.** Excluded from the matrix via `CARVE_OUT_KEYS`
rather than satisfying its precondition. `docker compose config` rejects a
missing env file as a fact about the *reading host's* filesystem, not the
document — compose2pod's script runs elsewhere, where the file is checked out
at run time, and the decision explicitly says a host-dependent verdict cannot
bind. `${VAR:?msg}` carries the same host-dependency, but no probe shape
produces a `${...}` reference, so it never needed separate handling.

**Two small open items folded in**, per review:

- `tests/test_cli.py::TestYaml11IntBoundaryForms` — `007`, `0x1A`, `0b101`,
  `1:20` still resolve as `int` after the 2026-07-15.01 float-resolver rebuild
  (it touches the same implicit-resolver table).
- `tests/test_parsing.py::test_build_mapping_rejects_non_string_key` —
  `_validate_build`'s `require_string_keys` guard on `build`'s own top-level
  keys (distinct from its unread contents) had no dedicated test.

## Non-goals

- No baseline, allow-list, or `xfail` mechanism — the maintainer explicitly
  ruled this out. A probe failure means a real gap; the gate gets fixed, not
  tolerated.
- Not exhaustive on multi-key interaction beyond the hand-authored corpus —
  the matrix probes one key at a time on one service.

## Testing

`just test-conformance`: **666 probes, 0 hard violations.** 554 both-reject,
110 both-accept, 2 over-reject (`sysctls: ["a"]`, `volumes: ["a"]` — both
already catalogued in `planning/deferred.md`). `just test-ci` still passes at
100% line coverage with the conformance suite deselected (`addopts`) and
omitted from coverage (`run.omit`). `just lint-ci` clean. New `conformance`
CI job in `.github/workflows/_checks.yml`, mirroring `integration` (no
daemon setup needed — `docker compose config` doesn't touch one).

## Risk

- **Docker version drift.** The harness tracks whatever `docker compose`
  version the CI runner ships; a Compose spec change could flip a verdict.
  Named as a revisit trigger in the decision itself.
