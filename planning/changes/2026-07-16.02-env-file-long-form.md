# Design: env_file long-form

## Summary

`env_file` accepts a string or list of strings today; Docker also accepts a
**mapping** entry `{path, required, format}` inside the list. This change accepts
that long-form: `path` emits the same `--env-file` flag the string form does,
`format` is validated then ignored (compose2pod hands the raw file to podman, so
Compose's parser directive can't change the flag), and `required: false` is
honored with an in-position run-time `[ -f path ]` guard so the flag is dropped
when the optional file is absent. The common path (string form, `required: true`)
renders byte-identically to today.

## Motivation

Long-form `env_file` is a cataloged over-reject (`planning/deferred.md`; the
conformance harness reports the `env_file_long_form` corpus document). Docker
accepts `env_file: [{path: app.env, required: false}]`; compose2pod's
`_validate_string_or_string_list` requires every entry to be a string. The
modern Compose spec and templating tools (`yq`, Helm-style generators) emit the
mapping form, so this is a realistic user-facing gap.

## Design

### Grammar (measured vs `docker compose config` v5.1.2)

- `env_file`: a string, or a **list** whose entries are each a string **or** a
  mapping. A bare mapping (`env_file: {path: x}`) is rejected — Docker: "env_file
  must be a string".
- Mapping is a **strict schema**: `path` (required, string); `required`
  (optional bool, default `true`); `format` (optional — `raw` is the *only*
  accepted explicit value; `dotenv`/`env`/`default`/empty are all rejected).
  Any other key rejects (`additional properties`).
- `required` accepts the quoted-boolean form via `values.is_bool_like`
  (`2026-07-16.01`) — measured: `required: "no"` accepted, `required: 5`/`banana`
  refused.

### Validation (`parsing.py`)

`_validate_env_file` stops sharing `_validate_string_or_string_list` (which
`tmpfs` keeps). New logic: accept a string; accept a list, checking each entry is
a string or a strict-schema mapping (`require_string_keys`; `path` present and a
string; `required` via `is_bool_like`; `format` in `{"raw"}`; no unknown key).

### Emit (`emit.py`) — the easy two

- **`path`** → `--env-file <project_dir/path>`, identical to the string form.
  `required: true` and string entries render **unconditional inline**, unchanged.
- **`format`** → validated, never emitted. compose2pod delegates env-file reading
  to podman's `--env-file` (podman does its own parsing); Compose's `format`
  governs *Compose's* parser, which compose2pod never runs — so `raw` vs default
  cannot change the emitted flag. Accept-and-ignore, documented (not a divergence:
  the flag and podman's behavior are the same either way).

### Emit — `required: false` (the real work)

Honored via an **in-position** run-time guard, preserving podman's
later-`--env-file`-overrides-earlier precedence. For each `required: false`
entry, prelude lines before the service's `podman run`:

```sh
c2p_envfile_<i>=""
[ -f "<path>" ] && c2p_envfile_<i>="--env-file=<path>"
```

and, at the entry's list position in the run token stream, the raw token
`${c2p_envfile_<i>:+"$c2p_envfile_<i>"}`:

```sh
podman run -d --pod p --name p-app --env-file /proj/base.env ${c2p_envfile_0:+"$c2p_envfile_0"} ... img
```

`${var:+"$var"}` is POSIX (present → the single quoted word; absent → nothing).
The `--env-file=PATH` **equals form** keeps it one shell word even for a path with
spaces, preserving compose2pod's no-word-splitting/no-injection discipline. The
path may itself contain `${VAR}` (env_file paths are `Expand` tokens today); the
guard's `[ -f … ]` test and the assignment use the same rendered path, so
run-time interpolation works identically in the test and the flag.

### New emit machinery (inert unless a `required: false` entry exists)

1. A **guarded token** variant (a frozen dataclass beside `Expand`) carrying the
   path; `_render` renders it raw as `${name:+"$name"}`, and its path still feeds
   `_collect_vars`/`variable_names` so the run-time-variable list stays complete.
2. **Per-service prelude lines**: the run renderer yields the assignment lines for
   its guarded entries; `_plan`'s per-service branch (`_emit_target` and the dep
   `-d`/`--rm` lines) emits them just before the `podman run` line. Each guard's
   variable name is a shell-inert identifier unique per service+entry.

## Non-goals

- **`format=raw` semantics** — compose2pod doesn't parse env files, so `raw` is
  accept-and-ignore. Not a divergence.
- **Reordering optional files** — in-position emission preserves precedence; no
  hoisting of optional entries.
- **Moving `env_file` into `SERVICE_KEYS`** (the registry-unification arc) — out
  of scope; env_file needs `project_dir`, which `KeySpec.emit` does not pass.

## Testing (TDD; Docker + podman oracles)

- **parsing**: accept string, list-of-strings, list-with-mapping; strict schema
  rejects (missing `path`, unknown key, bad `format`, non-bool-like `required`,
  bare mapping); `required: "no"` accepted; `tmpfs` still string-or-list.
- **emit**: `required: true` and string entries render unconditional inline
  (existing snapshots unchanged); `required: false` renders the prelude +
  `${var:+"$var"}` at the correct list position; multiple mixed entries preserve
  order; a `${VAR}` in the path interpolates in both the `[ -f ]` test and the flag.
- **run-time variables**: a `${VAR}` in a `required: false` path is still reported
  by `referenced_variables` (guarded token wired into `_collect_vars`).
- **podman parity** (integration): confirm podman accepts `--env-file=PATH`
  (equals form) and that an absent `required: false` file leaves the container
  running with no error while a present one lands its vars. Verify the equals form
  against real podman *before* committing the mechanism.
- **conformance**: the `env_file_long_form` corpus document flips
  over-reject → both-accept (no code change to the harness beyond this).
- **promotion**: `architecture/supported-subset.md` (env_file section) and
  `planning/deferred.md` (remove the long-form `env_file` bullet).
- `just test-ci` @ 100% coverage, `just lint-ci`, `just check-planning`.

## Risk

- **podman rejects the `--env-file=PATH` equals form** (low × high): breaks the
  guard. Mitigation: measure against real podman in TDD before committing; fall
  back to a `set --` accumulator if the equals form fails.
- **Prelude-line plumbing touches `_plan`/`_emit_target`** — a structural emit
  change (low × med): mitigated by snapshot tests proving the no-`required:false`
  path is byte-identical, plus new tests for the guarded path.
- **`referenced_variables` drift** if the guarded token isn't wired into
  `_collect_vars` (low × med): a direct test that a `${VAR}` in a `required:
  false` path is reported.
