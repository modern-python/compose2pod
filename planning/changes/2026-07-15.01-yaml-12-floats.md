# Design: Read YAML floats the way Docker reads them

## Summary

PyYAML implements **YAML 1.1**, whose float grammar requires a dot — a bare
`1e3` has none, so PyYAML leaves it the plain string `"1e3"`. Docker Compose
parses **YAML 1.2**, where an exponent alone is enough to make a float, so to
Docker `1e3` is `1000.0`. `_build_yaml_loader` (`compose2pod/cli.py`) already
carries a 1.2-style boolean resolver for exactly this class of divergence
(`2026-07-14.09`); this change gives it the sibling float resolver.

## Motivation

Measured against live `docker compose config` v5.1.2, feeding the same YAML
text to both oracles, one loader gap produces six wrong verdicts on the
resource keys:

| YAML | Docker | compose2pod before this change |
|---|---|---|
| `cpuset: 1e3` | **REJECT** (`cpuset` must be a string; Docker sees a float) | ACCEPT (sees the string `"1e3"`) — **a false green** |
| `oom_score_adj: 1e3` | ACCEPT (float 1000.0, a whole number) | reject |
| `cpu_shares: 1e3` | ACCEPT | reject |
| `cpu_quota: 1e3` | ACCEPT | reject |
| `cpu_period: 1e3` | ACCEPT | reject |
| `pids_limit: 1e3` | ACCEPT | reject |

`planning/decisions/2026-07-14-docker-rejection-parity.md`: **Docker rejects
⇒ compose2pod must reject, no exceptions.** The `cpuset` row is a hard-rule
violation — accepting a document Docker refuses emits a script for a file
that is already broken upstream, the exact false green the gate exists to
catch.

**No validator can fix this.** A *quoted* `"1e3"` really is rejected by
Docker on those keys (it's a string; `strconv.ParseInt`/the string-cast field
fails). Only the loader can tell a bare `1e3` from a quoted `"1e3"` apart —
by the time a validator sees the parsed value, the spelling is gone.

## Design

`_build_yaml_loader` drops PyYAML's YAML 1.1 float resolver (alongside the
bool resolver it already drops) and installs a YAML 1.2 core-schema one:

```
_YAML_12_FLOAT = (
    r"^[-+]?(?:[0-9]+\.[0-9]*|\.[0-9]+)(?:[eE][-+]?[0-9]+)?$"  # dotted mantissa, optional exponent
    r"|^[-+]?[0-9]+[eE][-+]?[0-9]+$"                            # undotted mantissa, mandatory exponent
    r"|^[-+]?\.(?:inf|Inf|INF)$"
    r"|^\.(?:nan|NaN|NAN)$"
)
```

Two branches, not one, because YAML 1.2's core schema treats "has a dot" and
"has an exponent" as independent ways to become a float — either is
sufficient, but *neither* present must fall through to the int resolver.
That's the correctness constraint driving the shape: a plain integer like
`123` has no dot and no exponent, so it cannot match either branch and stays
an int, exactly as before. `1_000` is likewise unaffected — the int resolver
is untouched, and the new float regex has no digit-grouping-underscore
branch to accidentally catch it.

A quoted `"1e3"` never reaches implicit resolution at all: PyYAML tags a
quoted scalar `str` directly, the same mechanism that already keeps a quoted
`"on"` a string under the 1.2 bool resolver. That's the whole point of the
fix — only the *bare* spelling needs to change.

## Non-goals

- Not a general YAML 1.2 port. Only the float resolver moves, exactly as
  `2026-07-14.09` moved only the boolean resolver.
- No digit-grouping underscore or sexagesimal (`:`) support in the float
  grammar — YAML 1.2's core schema has neither, and PyYAML's own int/null/
  timestamp resolvers are untouched.
- Not changing `_render_scalar` or any value grammar in `values.py` — the six
  rows above are resolved entirely by the loader now seeing the correct
  Python type; `validate_string`/`validate_count`/`validate_integer` already
  handle a native `float` correctly and needed no change.

## Testing

`just test-ci` at 100%. The loader is tested directly in
`tests/test_cli.py::TestYaml12Floats` (bare `1e3`/`-1e3`/`1E3` load as float;
`1.5`/`.5`/`.inf`/`-.inf`/`.nan` still load as float; a bare `123` and a
digit-grouped `1_000` still load as int; a quoted `"1e3"` stays a string) and
end-to-end through the CLI for the two representative rows: `cpuset: 1e3` is
rejected ("must be a string"), `oom_score_adj: 1e3` is accepted. All six rows
from the motivation table were re-measured against `docker compose config`
v5.1.2 after the fix and now match.

## Risk

- **A document relying on `1e3` meaning the string `"1e3"`.** It never meant
  that to Docker, so such a file was already being converted wrongly here;
  this makes it match the tool it replaces.
