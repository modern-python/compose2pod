# Design: implicit default network and named-volume reference parity

## Summary

A whole-branch review of `planning/decisions/2026-07-14-docker-rejection-parity.md`'s
hard rule found three surviving divergences. One is a real over-rejection:
`_validate_network_references` required `networks: [default]` to be declared
top-level, but `default` is Docker's implicit network and never needs
declaring. The other two are false greens: compose2pod had no volume analogue
of `_validate_network_references` at all, so a service naming an undeclared
NAMED volume was silently accepted; and `SUPPORTED_TOP_LEVEL_KEYS` listed
`name`/`version` without ever checking their value's type, so `name: 123` and
`version: 123` were accepted too. All three are fixed here. Two further false
greens (the non-target dependency graph) are deliberately deferred — see
`planning/deferred.md`.

## Motivation

Measured, same YAML both oracles (`docker compose config` v5.1.2):

- `networks: [default]`, no top-level `networks:` block — Docker ACCEPT;
  compose2pod REJECTed ("undefined network default") — the over-rejection.
- `networks: [host]` / `networks: [none]` / `networks: [mynet]`, all
  undeclared — Docker REJECT (unchanged; only `default` is implicit).
- `volumes: [data:/var]`, no top-level `volumes:` block — Docker REJECT
  ("refers to undefined volume data"); compose2pod ACCEPTed — the false green.
- `volumes: [data:/var]` with `volumes: {data:}` declared — Docker ACCEPT
  (unchanged).
- `volumes: [./host:/var]` / `volumes: [/abs:/var]` / `volumes:
  [/container_only]` — Docker ACCEPT, no declaration needed for a bind mount
  or an anonymous volume (unchanged).
- `volumes: [data:/var]` with `volumes: {data: {external: true}}` — Docker
  ACCEPT; `external: true` still counts as a declaration for the reference
  check.
- `volumes: [${VOLNAME}:/var]` — host-state-dependent either way: Docker
  resolves the variable first and classifies the *result* (a resolved bind
  path needs no declaration; a resolved bare identifier does) — a fact about
  the reading shell, not the document, so this is exempted the same way every
  other `${VAR}`-carrying grammar in `parsing.py` already is.
- `name: 123` / `version: 123` — Docker REJECT ("name must be a string" /
  "version must be a string"); compose2pod ACCEPTed. A bare `name:`/`version:`
  (null) is refused the same way.
- `name: valid-name`, `version: "3.8"` — Docker ACCEPT (unchanged).

## Design

**`compose2pod/parsing.py`, `_validate_network_references`:** `declared = set(top_networks
or {})` becomes `declared = set(top_networks or {}) | {"default"}` — `default`
is always implicitly declared, whether or not the document declares it
explicitly (an explicit `networks: {default: {...}}` is legal too, and now a
no-op against the set union).

**New `_named_volume_source(volume: str) -> str | None`:** the bind-vs-named
classification, mirroring `emit.py`'s `_volume_flags` (source starts with `.`
or `/` → bind mount, no colon → anonymous, otherwise → a named volume's bare
identifier).

**New `_validate_volume_references(compose, services)`:** mirrors
`_validate_network_references` one level down. For each service's volume
entry, `_named_volume_source` extracts the named-volume source (or `None` for
a bind mount / anonymous volume, which need no declaration); a `${VAR}`-
carrying source is exempted (`values.has_variable`); everything else must
appear in the top-level `volumes:` block. Runs where `_validate_network_references`
used to be the sole owner of the top-level-mapping-shape check for its own
block — `_validate_volume_references` now owns that same check for `volumes`
(previously owned by `_validate_volume_definitions`), and `_validate_volume_definitions`'s
own duplicate check is dropped, exactly mirroring how `_validate_network_definitions`
already relies on `_validate_network_references` running first.

**New `_validate_top_level_scalar_strings(compose)`:** one `isinstance` check
per key over `("name", "version")`, called from `validate()` right after
`_reject_null_top_level_blocks`. A null value falls through the same
`isinstance(value, str)` check (never a separate null case), reproducing
Docker's identical "must be a string" verdict for both a wrong-typed value
and a bare `name:`.

## Non-goals

The two non-target dependency-graph false greens (`depends_on` on an
out-of-closure service, a cycle among out-of-closure services) — deliberately
deferred, catalogued in `planning/deferred.md` with a revisit trigger, not
fixed here; they fight compose2pod's closure-scoped validation design rather
than being an oversight.

Matching Docker's message text verbatim — only a clean `UnsupportedComposeError`
naming the right key/value is required.

## Risk

Low. Fix 1 only widens acceptance (the over-rejection direction the parity
rule does not forbid); Fixes 2 and 3 only narrow acceptance, and only for
documents Docker itself already refused, so no compose2pod user could have
run either document through `docker compose` successfully either.
