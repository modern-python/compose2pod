# Design: accept quoted booleans on every boolean field

## Summary

Docker casts a *string* on any boolean field through YAML-1.1 boolean rules
(`read_only: "yes"`, `privileged: "on"`, `internal: "off"` all run); compose2pod
requires a real YAML boolean and refuses every quoted form. This is the largest
cataloged over-reject in `planning/deferred.md`. This change adds one shared
primitive to `values.py` — `is_bool_like` (the accept-test) and `as_bool` (the
coercion) backed by the two measured spelling sets — and routes all six boolean
validators through it, closing the family uniformly. `keys._bool`'s emit coerces
before its truthiness test so a quoted `"false"` cannot leak a set flag.

## Motivation

Every boolean field in the project re-implements the same strict-bool refusal,
and prior decisions (`_validate_build_bool`, `_validate_definition_bool`
docstrings) kept build/definition strict *for consistency with the six service
keys*. So the over-reject is one family, not six unrelated gaps. It is the item
most likely to hit a real user: anything that round-trips YAML (`yq`, Helm-style
generators, quote-everything house styles) emits `tty: "true"`. The conformance
harness reports these as `over-reject`, so they are already visible.

Measured against `docker compose config` v5.1.2 — accepted string set is exactly
the YAML-1.1 boolean spellings, uniform across all six families, standard
true/false split:

- **true**: `y Y yes Yes YES true True TRUE on On ON`
- **false**: `n N no No NO false False FALSE off Off OFF`
- **refused**: `t f 1 0 banana ""` (and any other string, any non-string)

compose2pod's YAML-1.2 reader (`read.py`) demotes bare `on`/`off`/`yes`/`no` to
strings; this field-level cast is what re-accepts them, matching Docker.

## Design

**Primitive (`values.py`)**, beside `has_variable`/`is_number`:

```python
_BOOL_TRUE  = frozenset({"y","Y","yes","Yes","YES","true","True","TRUE","on","On","ON"})
_BOOL_FALSE = frozenset({"n","N","no","No","NO","false","False","FALSE","off","Off","OFF"})

def is_bool_like(value):   # the accept-test every bool validator uses
    return isinstance(value, bool) or (isinstance(value, str) and (value in _BOOL_TRUE or value in _BOOL_FALSE))

def as_bool(value):        # coerce a validated bool-or-bool-string to a real bool
    return value if isinstance(value, bool) else value in _BOOL_TRUE
```

`is_bool_like` guards the `str` check so an unhashable dict/list value cannot
crash the `in` test. Accept and coerce read the same two sets, so drift is
unrepresentable — the single-seam pattern the registry keys already use.

**Accept-side (six validators, keep each existing `${VAR}` carve-out):** swap
`isinstance(value, bool)` for `values.is_bool_like(value)` in
`keys._validate_bool` (`init`, `read_only`, `privileged`, `oom_kill_disable`),
`parsing._validate_bool` (`tty`, `stdin_open`), `parsing._validate_build_bool`
(`no_cache`, `pull`, `privileged`), `parsing._validate_definition_bool`
(`internal`, `attachable`, `enable_ipv6`), the boolean branch of
`parsing._validate_definition_external`, and `graph._validate_depends_on_flag`
(`restart`, `required`). The three validators that carry a `has_variable` guard
keep it unchanged.

**Emit (one site):** `keys._bool` emit `[flag] if value else []` →
`[flag] if values.as_bool(value) else []`. This is the whole trap: a quoted
`"false"` is a truthy Python string and would otherwise emit the flag. Every
other boolean field is validate-only — `tty`/`stdin_open` and `depends_on`'s
`restart`/`required` are ignored at emit, and the network/volume definition
bools are validated for parity but never consumed in the single-pod model — so
accepting the string form there is inert downstream. Audit confirmed `keys._bool`
is the only value-truthiness read of a boolean field.

## Non-goals

- **`${VAR}` on an emitting bool** (`init: ${X}`): stays refused. A static
  script cannot conditionally emit a flag from a run-time variable, so it is a
  genuine can't-express limit, not this over-reject. This change is literal
  quoted strings only; every site's existing `${VAR}` behavior is preserved.
- **Unifying the two `_validate_bool` wrappers** (`keys` and `parsing`): their
  error messages differ; both route through `is_bool_like` internally, but the
  wrappers stay to preserve messages. No message-parity requirement — Docker
  parity is accept/reject only.
- **Changing refusal messages**: `"must be a boolean"` stays; only the accept
  set widens.

## Testing

TDD, Docker as oracle — failing tests first.

- **Primitive unit tests** (`tests/test_values.py`): every accepted spelling →
  correct `as_bool`; `is_bool_like` true for each; `t`/`f`/`1`/`0`/`banana`/`""`,
  a non-string, and an unhashable `dict`/`list` → `is_bool_like` False, no crash.
- **One accept + one reject per family** across `test_keys`, `test_parsing`,
  `test_graph`: e.g. `read_only: "yes"` accepted, `read_only: "banana"` refused;
  same for a build bool, a definition bool, `external`, and `depends_on.required`.
- **The trap regression** (`tests/test_emit.py`): `read_only: "false"` /`"no"`
  /`"off"` emit no `--read-only`; `"true"`/`"yes"`/`"on"` emit it.
- **Conformance** (`just test-conformance`): the affected cells flip
  over-reject→conformant; update the expected snapshot.
- `just test-ci` at 100% line coverage, `just lint-ci` clean, `just check-planning`.

Promote in the same PR: `architecture/supported-subset.md` (drop the strict-bool
limitation prose) and `planning/deferred.md` (remove the quoted-boolean bullet
and its Task-12 addendum).

## Risk

- **A downstream value-truthiness read of a boolean field beyond `keys._bool`**
  (low × high): would silently misbehave on a quoted `"false"`. Mitigated by the
  audit above (grep of every boolean key's consumers) and the trap regression
  test; the definition bools have no consumer at all.
- **A conformance cell that does not flip as expected** (low × med): the
  snapshot diff surfaces it before merge; each family was measured directly.
- **Coverage gap on a new `values` branch** (low × low): the reject/unhashable
  cases each have a direct assertion.
