# Design: networks/volumes definition schema and depends_on sub-key schema

## Summary

A controller sweep after `2026-07-15.11` (whose own summary claimed the
hard rule was "believed closed") found three more nested false-green
classes, all in regions compose2pod deliberately ignores at emit time but
Docker still validates — the same "ignored but still validated" stance
`build` has always had. Closed with the established strict-key-schema
pattern (`_DOCKER_BUILD_KEYS`, Task 9): a top-level `networks`/`volumes`
definition's own body, and a `depends_on` long-form entry's sub-keys.

## Motivation

`_sweep_document` and `_validate_network_references` were explicit that
top-level `networks`/`volumes` block *contents* were "never read" (`build`'s
own precedent), and `graph.depends_on` only ever extracted `condition` from
a long-form entry, silently accepting anything else. All three are
document-wide checks Docker enforces regardless of whether compose2pod uses
the value. Measured against live `docker compose config` v5.1.2, same YAML
text to both oracles:

- `networks: {mynet: somestring}` (non-mapping), `{mynet: {bogus: 1}}`
  (unknown key), `{mynet: {driver: 123}}` (wrong type) — all REJECT; we
  ACCEPTed all three.
- `volumes:` had the identical three false greens, plus one more: the
  top-level `volumes:` block's own *shape* was never checked at all (only
  `networks`'s was, via `_validate_network_references`) — `volumes:
  somestring` was previously accepted outright.
- `depends_on: {db: {condition: service_started, bogus: 1}}` — REJECT
  ("additional properties 'bogus' not allowed"); we ACCEPTed it.
- `depends_on: {db: {restart: 5}}` — REJECT ("must be a boolean or
  string"); we ACCEPTed it (the key was never read).

## Design

**Top-level `networks`/`volumes` definition schema** (`compose2pod/parsing.py`):
the complete key sets were measured by probing every candidate key (`cidr`,
`internal_ipv6`, `scope`, `priority`, and Docker's own real key set)
individually against live `docker compose config` v5.1.2. Networks: 9 keys
— `driver`, `driver_opts`, `external`, `name`, `labels` (shared with
volumes), plus `ipam`, `internal`, `attachable`, `enable_ipv6`
(network-only; each measured absent and refused as an unknown key on a
volume definition). Volumes: the shared 5. `external` is a boolean, or a
mapping with an optional string `name` (deprecated spelling, still
accepted, measured); a bare string is refused outright — Docker casts it as
a boolean interpolation target instead of a plain string. `ipam` is the one
key validated two levels deep: `driver` (string), `config` (a list of
per-subnet mappings — `subnet`/`ip_range`/`gateway`, all strings, and
`aux_addresses`, a mapping of strings), and `options` (a mapping of
strings — measurably *not* the number-or-string union `driver_opts` has).

New per-key validator factories (`_validate_definition_string`,
`_validate_definition_bool`, `_validate_definition_driver_opts`,
`_validate_definition_labels`, `_validate_definition_external`,
`_validate_definition_ipam`) parameterized on a `label` ("network"/
"volume"), since the existing per-key validators in this module hardcode a
`"service {name!r}"` message prefix that would misname a network or volume.
Two key-set dicts (`_NETWORK_DEFINITION_KEYS`, `_VOLUME_DEFINITION_KEYS`)
feed a shared `_validate_top_level_definition`, mirroring `_validate_build`'s
own strict-schema loop. `_validate_network_definitions`/
`_validate_volume_definitions` run from `validate()` right after
`_validate_network_entries`; `networks`/`volumes` join the
`_sweep_document` deep-sweep loop (previously `secrets`/`configs` only) so a
non-string key inside a definition fails clean instead of silently passing
through the region that used to be "never read."

`_validate_definition_bool` and the boolean branch of
`_validate_definition_external` apply `values.has_variable`'s carve-out —
measured, Docker resolves a `${VAR}` reference and casts it to boolean at
*read* time ("error while interpolating ... failed to cast to expected
type"), so its verdict is a fact about the reading shell, not the document,
the same carve-out `_validate_build_bool` already has. A literal quoted
string (`internal: "true"`) is still refused, joining the existing
six-top-level-boolean-key limitation in `planning/deferred.md`.

**`depends_on` sub-key schema** (`compose2pod/graph.py`): new
`_DEPENDS_ON_SUB_KEYS = {"condition", "restart", "required"}` and
`_validate_depends_on_flag` (the same has_variable-carved-out strict bool as
above), wired into a new `_depends_on_entry_condition` helper split out of
`depends_on` itself to keep that function under the complexity linter's
threshold. Runs for every long-form entry: unknown keys raise, `restart`/
`required` get the same boolean grammar. `condition`'s own existing
type/membership checks are untouched.

## Non-goals

**`depends_on`'s `condition` becoming a required key.** Measured, unplanned
finding: `docker compose config` v5.1.2 refuses a long-form entry with no
`condition` at all (`{db: {}}`, or `{db: {restart: true}}`) —
"missing property 'condition'" — even though the short (list) form still
defaults silently to `service_started`. This directly contradicts this
project's own `graph.depends_on` (`spec.get("condition", "service_started")`),
`architecture/supported-subset.md`, and a committed integration test
(`tests/integration/test_depends_on_default.py`) asserting the opposite.
Reproduced directly against the installed `docker compose config` v5.1.2 (not
inferred), independent of any brief. This is exactly the revisit trigger
`2026-07-14-docker-rejection-parity.md` names ("Docker's own validation
changes such that a construct compose2pod correctly accepts starts being
refused") and forces a design choice — which Compose version's behavior the
invariant tracks — left to a follow-up decision rather than decided here
unilaterally; enforcing it would break the existing integration test and the
documented default-condition behavior. Not implemented in this change.

## Testing

**Unit**: `tests/test_parsing.py` (`TestNetworkDefinitionSchema`,
`TestVolumeDefinitionSchema` — non-mapping, null, unknown key, extension
key, non-string key, every field's type, `ipam` two levels deep, the
`${VAR}`-vs-quoted-string distinction on every boolean field, and
`test_top_level_volumes_non_mapping_rejected` for the previously-unchecked
outer shape); `tests/test_graph.py` (`TestDependsOn` — unknown key,
`restart`/`required` type, `${VAR}` carve-out, `x-` extension). Confirmed
every reject-side test failed (`DID NOT RAISE`) against pre-fix code, then
passed after the fix. Two existing tests
(`TestSweepSkipsUnreadRegions::test_top_level_{networks,volumes}_contents_non_string_key_*`)
flipped from accept-with-warning to reject, since the regions they probe are
no longer "never read." `just test-ci`: 1224 passed, 100% line coverage.

**Conformance** (`tests/conformance/corpus/`): 10 new files —
`network_definition_{non_mapping,unknown_key,driver_wrong_type,
ipam_config_unknown_key,external_bad_string}`, `volume_definition_{non_mapping,
unknown_key,external_wrong_type}`, `depends_on_{unknown_sub_key,
restart_wrong_type}`. All 10 confirmed failing (`AssertionError: docker
compose config REJECTS this document but compose2pod ACCEPTS it`) against
pre-fix code (`git stash` on `parsing.py`/`graph.py` only), then passing.
`just test-conformance`: 787 selected, 783 passed, 4 skipped, **0 hard
violations**. 12 over-rejections, identical set to `2026-07-15.11`'s
(quoted booleans, `sysctls`/`volumes` list-of-string edges, `env_file`/
duration host-state carve-outs) — none newly introduced, since the
generated matrix only probes top-level service keys and none of this
change's new fields are matrix-covered (nested positions, like the `networks`
entry schema before it).

`just lint-ci` and `just check-planning` both clean.

This closes the three remaining nested false-green classes found by the
post-`2026-07-15.11` controller sweep. No further nested region is known to
be unvalidated; the whole-document hard rule
(`accepted(compose2pod) ⊆ accepted(docker)`) is believed closed again,
with the same caveat `2026-07-15.11` carried and this change's own
non-goal reinforces: "believed" tracks what has been swept, not a proof.

## Risk

Low — both schemas only narrow what was previously accepted unconditionally
or entirely unchecked, the direction the parity rule requires. The
`depends_on` `condition`-required finding is flagged, not acted on, to avoid
an unreviewed behavior change; see Non-goals.
