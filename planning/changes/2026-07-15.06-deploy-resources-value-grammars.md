# Design: Per-field grammars for `deploy.resources` limits/reservations

## Summary

`compose2pod/resources.py`'s `_check_number` (built on `keys.is_number`, which
accepts any number or any string unconditionally) was the single gate for
`deploy.resources.limits.{cpus,memory,pids}` and
`deploy.resources.reservations.memory`. It let through documents `docker
compose config` rejects — `limits: {memory: ""}`, `limits: {memory: 512}`,
`limits: {cpus: abc}`, and more — a direct breach of the docker-rejection
parity rule (`planning/decisions/2026-07-14-docker-rejection-parity.md`).
`_check_number` is gone; each field now validates against its own grammar
drawn from `compose2pod/values.py`, the same module the top-level
`cpus`/`mem_limit`/`pids_limit`/`oom_score_adj` keys already use.

## Motivation

Tasks 1–5 (2026-07-14) tightened every *top-level* service key's value
grammar and built a conformance harness to hold the line. That harness's
generated matrix probes one key at a time on a single service — it cannot
reach a nested position like `deploy.resources.limits.memory`, so these four
fields kept the pre-tightening loose check indefinitely. Measured against
live `docker compose config` v5.1.2 (same YAML text fed to both oracles):
`limits.memory`/`reservations.memory` are typed as a plain Go **string**
field, not the size-or-string union `mem_limit`/`mem_reservation` are — a
*native number* is refused (`memory: 512` ❌, `1.5` ❌), which
`keys.is_number` waved through. `limits.cpus` empty-string and non-numeric
strings were waved through the same way. `limits.pids` had the same gap for
fractional values (`pids: 0.5`, `pids: "0.5"` accepted by `is_number`,
refused by Docker).

## Design

**`values.validate_size` gained a `string_only: bool = False` keyword**,
mirroring the existing `allow_fractional` style rather than a hand-rolled
parallel check. With `string_only=True`, the native-number branches
(`_is_int`, the float branch) are skipped entirely — only the `${VAR}`
short-circuit and the string/size-regex branch remain reachable. Every
existing caller passes no `string_only` (defaults to `False`), so
`mem_limit`/`mem_reservation`/`mem_swappiness`/`shm_size` behave exactly as
before; their test suites re-ran unchanged and green.

**`resources.py`'s `_LIMITS` table gained a third column** — a per-field
validator — replacing the single `_check_number` call site:

```python
_LIMITS: dict[str, tuple[str, str, Callable[[str, str, Any], None]]] = {
    "cpus": ("--cpus", "cpus", values.validate_number),
    "memory": ("--memory", "mem_limit", _validate_memory_size),
    "pids": ("--pids-limit", "pids_limit", _validate_pids_count),
}
```

- `limits.cpus` → `values.validate_number` (identical grammar to the
  top-level `cpus` key — native float or numeric string; measured accept:
  `2`, `0.5`, `"0.5"`, `"2"`; reject: `""`, `abc`).
- `limits.pids` → `values.validate_integer(..., allow_whole_float=True)`
  (identical to `oom_score_adj` — accept `100`, `"100"`, `100.0`, `-1`;
  reject `0.5`, `"0.5"`, `"1e3"`).
- `limits.memory` / `reservations.memory` → `values.validate_size(...,
  string_only=True)` via a shared `_validate_memory_size` helper (both
  fields carry the identical grammar, re-measured by hand for
  `reservations.memory` rather than assumed from `limits.memory`).

`resources.py` no longer imports `keys.is_number` (dropped along with
`_check_number`; `keys.py`'s other caller, `parsing.py`'s healthcheck-scalar
check, is untouched).

`test_limit_value_bool_rejected`'s expected message moved from `"must be a
number or string"` to `"must be a size string"`, since `memory: True` now
fails the `string_only` branch's message rather than the deleted
`_check_number`'s.

## Non-goals

`reservations.cpus` and `reservations.devices` stay refused outright — no
podman equivalent, unaffected by this change. No change to the
conflict-detection logic (legacy-vs-`deploy.resources` overlap), the
null-block checks, or the YAML loader.

## Testing

**Unit** (`tests/test_values.py`, `tests/test_resources.py`): new
parametrized cases for `validate_size(string_only=True)` (native
number/bool rejected, size string accepted, non-size string rejected,
`${VAR}` still short-circuits) and for each of the three
`deploy.resources` fields (accept/reject pairs matching the measured
grammars). Confirmed red (`TypeError: unexpected keyword argument
'string_only'`, then `DID NOT RAISE`) before implementing, green after.
`just test-ci`: 970 passed, 100% line coverage.

**Conformance** (`tests/conformance/corpus/`): 7 new hand-authored
documents — `deploy_resources_limits_{memory_empty_string,
memory_non_size_string, memory_native_number, cpus_empty_string,
cpus_non_numeric_string, pids_fractional}.yaml` and
`deploy_resources_reservations_memory_empty_string.yaml` (7 files covering
the 5 violations from the task brief plus two more the same sweep turned up
— native-number `memory` and fractional `pids`, both real hard-rule breaches
under the old `is_number` gate, both already implied by the per-field
grammars above). Verified each fails (`AssertionError`: docker rejects,
compose2pod accepted) against the pre-fix code via `git stash`, and passes
after. `just test-conformance`: 745 selected, 741 passed, 4 skipped, 0
failed — 0 hard violations; 12 over-rejections, all pre-existing and
catalogued in `planning/deferred.md`, none touching `deploy.resources`.

`just lint-ci` and `just check-planning` both clean.

## Risk

Low. The change narrows what three previously-loose fields accept; it
cannot newly *accept* a document Docker rejects (the failure mode the parity
rule forbids), only newly refuse ones already broken upstream, or — in the
one bidirectional case (`string_only`) — nothing existing regresses, since
every prior `validate_size` caller keeps its default (`string_only=False`)
and was re-run unchanged. The main risk was assuming
`reservations.memory` shares `limits.memory`'s grammar without measuring it
directly — mitigated by re-measuring it by hand (empty string, non-size
string, native int, native float, and four accepted string forms) rather
than inferring it from the `limits.memory` result.
