# Change: tilde bind-mount volume-reference parity

**Lane:** lightweight â€” 2 source files, no new module, no public-API change,
straightforward tests.

## Goal

`volumes: [~/data:/var]` with no top-level `volumes:` block: `docker compose
config` v5.1.2 ACCEPTs it (a bind mount, expanded against the invoking user's
home directory). compose2pod REJECTs it ("refers to undefined volume
'~/data'"). Fix the over-rejection without weakening the undefined-named-
volume rejection Task 14 closed.

## Approach

`_named_volume_source`'s classifier tested "source does not start with `.` or
`/`" to decide "named volume, needs declaring" â€” true for a bare identifier
(`data`), but also true for every other bind-mount spelling that isn't a dot-
or slash-prefixed path: `~/data`, and (separately, not fixed here â€” see
Non-goals) a Windows drive path.

Replaced the startswith test with a positive match against
`stores.NAME_PATTERN` (`^[a-zA-Z0-9][a-zA-Z0-9_.-]*$`) â€” the identical
grammar Docker's own compose-go already applies to a secret/config *name*,
reused rather than re-derived, since parsing.py already imports `stores` (no
cycle: `stores.py` never imports `parsing`). Made public (`NAME_PATTERN`, no
leading underscore) since a second module now reuses it.

A source is now a NAMED VOLUME iff it matches this pattern; everything else â€”
`./rel`, `/abs`, `~/data`, `${VAR}` â€” is not a name at all, so none of them
need a declaration. The `${VAR}` case needed a separate `values.has_variable`
carve-out before; it doesn't now, since `$`/`{`/`}` are none of them pattern
characters â€” `_named_volume_source` already returns `None` for any
`${...}`-carrying source unconditionally, so the redundant `has_variable`
check was dropped from `_validate_volume_references`.

**Minor refactor:** `_validate_service_volumes` (the per-service shape check)
and `_named_volume_source` (the cross-document declaration check) both used
to re-derive the colon-form split independently. Extracted one
`_classify_volume(volume) -> ("anonymous" | "named" | "bind", source |
None)` helper both now call, so the split is defined exactly once.

## Non-goals

**Windows drive-letter volume source is NOT fixed here.**
`volumes: ["C:\data:/var"]` with no top-level declaration: Docker ACCEPTs
(measured â€” compose-go special-cases a leading `<letter>:\` so the drive
letter stays attached to the source, resolving to `{source: C:\data, target:
/var}`). compose2pod still REJECTs ("refers to undefined volume 'C'"),
unchanged by this fix: the shared colon-split (`source, _, _ =
volume.partition(":")`, in both `_classify_volume` here and `emit.py`'s
`_volume_flags`) splits on the *first* colon regardless, so for this entry
`source` is just the single letter `"C"` â€” itself a syntactically valid
`NAME_PATTERN` match, not the unparseable string a plain reading of "doesn't
match the pattern" would suggest. Closing this needs genuine Windows-drive
detection ahead of the split, in both call sites â€” a materially bigger change
than this task's grammar swap, and one that also touches real emission
(`emit.py`), not just the gate. Catalogued in `planning/deferred.md` under
"Forms Docker accepts that compose2pod does not yet parse" with a revisit
trigger; discovered while measuring this fix, pre-existing since Task 14
introduced the named-volume reference check, not introduced by this change.
