# Design: close the trailing-newline false green across value grammars

## Summary

Python's regex `$` matches not only at the end of a string but also *before a
trailing newline*. Every value grammar in `values.py` that validates via
`_REGEX.match(value)` is anchored with `$`, so a value carrying a trailing `\n`
— `mem_limit: "512m\n"`, reachable through an ordinary YAML block scalar
(`mem_limit: |`) — is accepted, while Docker rejects it. That is a violation of
the hard rule `accepted(compose2pod) ⊆ accepted(docker)`. This change anchors
each affected regex with `\Z` (true end of string only), closing the class.

## Motivation

Discovered while fixing the same flaw in `healthcheck.interval_seconds`
(`2026-07-16.03`, which the reviewer caught): `interval` was fixed there and the
systemic version scoped out to this dedicated change. Measured against
`docker compose config` v5.1.2, a trailing newline is a false green on:

- **`_SIZE`** (`validate_size`): `mem_limit`, `memswap_limit`, `mem_reservation`,
  `shm_size`, and `deploy.resources` memory.
- **`_STRICT_INT_STRING`** (`validate_count`): `cpu_shares`, `cpu_quota`,
  `cpu_period`, `pids_limit`.
- **`_DURATION`** (`validate_duration`): `stop_grace_period`, healthcheck
  `timeout`, `start_period`.
- **`_PORT`** (`validate_ports`): `ports`.

The float/int validators that went through the padded-scalar work
(`2026-07-15.16`) are already safe: `validate_number` (`cpus`),
`validate_integer` (`oom_score_adj`, `ulimits`), and `validate_native_number`
(`priority`) guard their coercion path with `value == value.strip()`, which a
trailing newline fails. The gap is only the four `.match`-based grammars, whose
`$` anchor never saw a strip guard. The flat conformance matrix's `padded-scalar`
probe (`" 512m "`) tests spaces, not newlines, so the class went uncaught.

## Design

**`compose2pod/values.py`** — change the trailing `$` to `\Z` in the four
regexes:

```python
_SIZE = re.compile(r"...\Z", re.VERBOSE)          # was ...$
_DURATION = re.compile(r"^[+-]?(?:[0-9]+(?:\.[0-9]+)?(?:ns|us|µs|ms|s|m|h))+\Z")
_STRICT_INT_STRING = re.compile(r"^[+-]?[0-9]+\Z")
_PORT = re.compile(r"...\Z", re.VERBOSE)          # was ...$
```

`\Z` matches only the true end of the string, so a trailing `\n` no longer
slips past. A newline-free value matches identically, so no Docker-valid value
is newly refused. The leading anchor stays `^` (a *leading* newline is already
refused: non-MULTILINE `^` matches only at string start). Internal whitespace was
already refused (no `\s` in any pattern).

**Scope is complete.** The only other `^...$` regexes in the package
(`stores.NAME_PATTERN`, `stores._ENV_NAME`, `emit.POD_NAME_PATTERN`) are used via
`.fullmatch`, which requires the *whole* string to match and so already rejects a
trailing newline; `graph._CONTAINER_NAME` is an unanchored Docker-style search
pattern. None is a `.match`-anchored value grammar, so none is affected.

## Non-goals

- **Adding `\s`/`strip` handling to the grammars** — the `\Z` anchor is the whole
  fix; the grammars already reject leading and internal whitespace.
- **A matrix `scalar-newline` shape** — a single probe value can't cleanly test
  heterogeneous grammars (`"512m\n"` is invalid on an int key for the *wrong*
  reason, so it wouldn't isolate the newline hole). Per-grammar guards instead.
- **`healthcheck.interval` / `_INTERVAL_DURATION`** — already fixed in
  `2026-07-16.03`.

## Testing

TDD, Docker as oracle.

- **Unit tests** (`tests/test_values.py`): `validate_size`, `validate_count`,
  `validate_duration`, and `validate_ports` each reject a trailing-newline value
  (`"512m\n"`, `"1024\n"`, `"5s\n"`, `"8080\n"`) with `UnsupportedComposeError`;
  a spot-check that the newline-free form still validates. Direct
  RED-before/GREEN-after.
- **Conformance corpus** (`tests/conformance/corpus/`): four documents, one per
  grammar family, using a double-quoted `"...\n"` scalar on a representative key
  (size → `mem_limit`, count → `cpu_shares`, duration → `stop_grace_period`,
  port → `ports`). Each is a hard-rule guard: the generic
  `test_corpus_document_obeys_the_rule` fails on the false green before the fix
  (Docker rejects, compose2pod accepts) and passes (`both-reject`) after.
- Promote `architecture/supported-subset.md` with a brief note that the value
  grammars reject surrounding whitespace and trailing newlines, matching Docker.
- `just test-ci` @ 100% coverage, `just lint-ci`, `just check-planning`,
  `just test-conformance`.

## Risk

- **`\Z` refuses a value Docker accepts** (low × high): only a trailing-newline
  value changes verdict, and Docker rejects every one of those — verified by the
  per-grammar corpus docs going `both-reject`. A newline-free value matches `\Z`
  exactly as it matched `$`, so no regression.
- **A grammar uses `$` mid-pattern, not only as the trailing anchor** (low × med):
  `_SIZE`/`_PORT` are multi-line `re.VERBOSE` patterns — the edit must target the
  final anchor only. Confirmed by reading each pattern; the unit tests cover the
  accept path so a mangled pattern fails loudly.
- **Another `.match`-anchored value grammar was missed** (low × med): the scope
  grep (above) found none outside the four; the name patterns use `.fullmatch`.
