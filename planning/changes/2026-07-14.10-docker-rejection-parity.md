# Design: Refuse every document Docker refuses

## Summary

compose2pod accepts 113 documents that `docker compose config` rejects. Each is
a false green: the developer's `docker compose up` fails locally while CI passes
on a script generated from the same file. This closes all 113 and adds the
harness that stops the next one, per
`decisions/2026-07-14-docker-rejection-parity.md`.

## Motivation

Measured, not assumed — 672 probes, both oracles, full detail in
`audits/2026-07-14-docker-rejection-parity.md`. (116 probes diverge; three are
the `env_file` host-dependent carve-out and are correct as they stand.) Three
code classes:

- **68** — the seven `IGNORED_SERVICE_KEYS` are warned about and then never
  looked at, so `restart: {a: {b: 1}}` and `ports: 3` sail through. "Ignored"
  was conflated with "unvalidated".
- **28** — `is_number` returns true for *any* string, so every resource key
  takes any value: `mem_limit: ""`, `cpus: somevalue`. Docker validates a real
  size/number grammar. The likeliest of the three to hit a real user
  (`mem_limit: 512`, no unit).
- **17** — structural keys: `build` (contents never read, so any shape passes),
  per-service `networks` naming an undeclared network, `dns_opt` accepting a
  string where Docker demands a list, and three empty-string cases.

Plus **3 documentation findings** where the tree contradicts the code (D1–D3 in
the audit), including a `supported-subset.md` that states both verdicts on
`entrypoint: null` and a `parsing.py` docstring asserting a Docker divergence
that measurement refutes.

The deeper problem is that nothing checked any of this. Five consecutive changes
(`2026-07-14.05`…`.09`) each hand-found one divergence — the shape of a missing
invariant.

## Design

**Ignored keys get a validator.** `IGNORED_SERVICE_KEYS` becomes a mapping from
key to shape validator, so a key can be *ignored at emit* while still *checked
at the gate* — the two were never the same thing. Five of the seven reach full
parity on a type check alone (Docker itself does not validate `restart`'s enum
or `stop_signal`'s content — measured). `ports` and `stop_grace_period` need a
grammar, below.

**Four value grammars**, in one new module, each returning a clean
`UnsupportedComposeError`:

```
size      512m | 1g | 1048576        -> mem_limit, memswap_limit,
                                        mem_reservation, mem_swappiness, shm_size
number    3 | 1.5 | "3"              -> cpus, cpu_*, pids_limit, oom_score_adj
duration  90 | 90s | 1m30s           -> stop_grace_period
port      8080:80/tcp | 3000-3005    -> ports
```

`cpuset` becomes string-only; `mem_reservation`/`mem_swappiness`/
`oom_score_adj` reject a float; `ulimits` bounds must cast to int.

**The one constraint that shapes every grammar:** a value carrying a `${VAR}`
reference is passed through **unvalidated**. Deferring interpolation to
script-run time is deliberate (`supported-subset.md`, Variable interpolation),
and a grammar that refused `mem_limit: ${MEM}` would break it. Docker never
meets this case because it interpolates first. So each grammar short-circuits on
a value containing a variable reference — the same test `shell.to_shell` already
applies.

**Structural fixes.** `build` must be a string or mapping (its contents stay
unread). A per-service network must be declared in the top-level `networks`
block, or be `default`. `dns_opt` must be a list. `image`/`container_name`
reject the empty string; `depends_on` rejects a bare string.

**The harness** — `tests/conformance/`, mirroring `tests/integration/` exactly:
a `conformance` marker, `addopts = -m 'not integration and not conformance'`, a
`run.omit` entry, an autouse fixture skipping when `docker` is absent, and its
own `ubuntu-latest` CI job (`docker compose config` needs no daemon, so there is
no setup step). It asserts the one-way rule — `not docker_ok ⇒ not c2p_ok` —
over two sources:

- a matrix generated from `SERVICE_KEYS | STRUCTURAL_KEYS | IGNORED_SERVICE_KEYS`
  crossed with the hostile shapes, so **a new registry key is probed the moment
  it is added** and the rule cannot decay as the subset grows;
- `corpus/*.yaml`, hand-authored, for what a single-key matrix cannot express:
  `extends` cycles, `depends_on` conditions, conflicting sysctls, nested
  `healthcheck`/`deploy`/store positions, top-level keys.

The probe runs the **real CLI pipeline** (loader → `resolve_extends` →
`validate` → `emit_script`), not `validate()` alone — the CLI runs both, and
`emit_script` catches what `validate` does not.

**The carve-out is honored by construction:** the harness materializes each
`env_file` and sets referenced variables before asking Docker, so Docker never
rejects for a reason that is a fact about the host rather than the document.

## Non-goals

- Not narrowing the subset. Every key compose2pod supports today it still
  supports; this only refuses values Docker refuses.
- Not validating what Docker doesn't. `restart`'s enum stays unchecked, because
  Docker accepts `restart: somevalue` too.
- Not resolving `${VAR}` at generation time (see above).
- Not fixing the two declared over-rejections (`sysctls: [a]`, `volumes: [a]`) —
  the soft rule permits them precisely because they are declared.

## Testing

`just test-ci` at 100%, and a new `just test-conformance` that must report **0
hard violations** across the full matrix — the gate this change exists to pass.
Each of the 113 gets a unit test at its own layer; the harness proves the set is
closed rather than merely sampled.

## Risk

- **Breaking, deliberately.** A document with `mem_limit: 512` or a malformed
  `ports` entry now raises where it previously converted. Every such document is
  one `docker compose` already refuses to run, so it only ever worked here by
  accident — and silently produced a script for a file that was already broken.
- **A grammar over-rejects.** The real hazard: a size/port grammar stricter than
  Docker's turns a soundness fix into a false alarm. The harness checks this
  direction too (it reports over-rejections), and every new one must be declared
  or fixed.
- **Sequencing.** This is one design but likely several PRs — harness first (it
  is the executable statement of the rule, and lands red), then the three code
  classes, then the doc fixes. The plan settles the split; each PR carries its
  own change file referencing this one.
