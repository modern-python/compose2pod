# Design: Surface the conformance harness's over-rejections

## Summary

`2026-07-15.02` built `tests/conformance/` and had `assert_rule` compute an
`'over-reject'` label for every document Docker accepts and compose2pod
refuses, but `test_matrix.py`/`test_corpus.py` only ever called `assert_rule`
for its assertion side effect and threw the returned label away. Combined
with a probe matrix whose twelve shapes were generic type-probes (`"str":
"somevalue"`) rather than the specific spellings Docker actually casts
(`"true"`, `1h30m`), the harness could neither surface an over-rejection nor,
for the compound-duration and long-form-`volumes` limitations, even reach one.
`planning/deferred.md`'s claim — "The conformance harness reports these as
`over-reject`, so they stay visible rather than forgotten" — was false. This
closes both gaps.

## Motivation

A promise a test suite makes about itself needs to be true, or every other
claim in `planning/deferred.md` is unverifiable by inspection. Concretely:
before this change, `just test-conformance` printed nothing about
over-rejections no matter how many existed, and the matrix's 12 shapes never
produced `tty: "true"` or `interval: 1h30m` — the two forms carrying the
largest catalogued limitation (30 measured cells across six boolean keys) and
the only nested one. A human running the suite had no way to check the
deferred list against reality.

## Design

**Reporting (`tests/conformance/conftest.py`).** A module-level
`pytest.StashKey[list[str]]` holds the run's over-rejection labels, created
empty in `pytest_configure` and read by `pytest_terminal_summary` — the
standard pytest pattern for state that must survive from a fixture teardown
to the end-of-session hook (a plain module global would work too, but the
stash is `Config`-scoped, so it can never leak state across two sessions
sharing one interpreter, e.g. `pytest-xdist` controller reuse). `assert_rule`
appends `request.node.nodeid` — the test id already carries the probed key and
shape (matrix) or the corpus filename (corpus), so no extra label plumbing is
needed to make an entry diffable against `planning/deferred.md`. Over-rejects
still never fail the build; the hook only prints, via
`terminalreporter.section(...)` / `.write_line(...)`, and is silent when the
list is empty (the normal case for `just test-ci`, which deselects
`conformance` entirely).

**Reaching the limitations.**

- `test_matrix.py` gains one shape, `"quoted-bool": "true"`. It alone
  surfaces all six boolean keys (`init`, `read_only`, `privileged`,
  `oom_kill_disable`, `stdin_open`, `tty`) as over-rejects, because
  compose2pod's `_validate_bool` rejects any non-`bool` uniformly —
  spelling never changes compose2pod's verdict, only Docker's. Adding
  `"yes"`/`"on"` as further shapes was considered and rejected: they would
  repeat the identical compose2pod-side rejection on the identical six keys,
  inflating the report 3x with no new signal. Verified directly against
  `docker compose config` v5.1.2 (matching `planning/deferred.md`'s measured
  version) before writing this: `read_only: "true"`, `"yes"`, and `"on"` all
  exit 0.
- Two new corpus files reach `healthcheck.interval`, which a flat
  one-key-per-service matrix structurally cannot: `1h30m` (compound) and `1h`
  (bare hour) — `planning/deferred.md` names both explicitly. Verified both
  accepted by `docker compose config` and refused by
  `healthcheck.interval_seconds` before adding.
- One new corpus file reaches long-form `volumes` (a `type`/`source`/`target`
  mapping) — the matrix's `list-of-map` shape (`[{"a": 1}]`) cannot stand in
  for it, since Docker validates the mapping's own keys and rejects an
  arbitrary one. Verified accepted by Docker, refused by
  `_validate_service_volumes` ("only short volume syntax is supported").

No shape or corpus file was added for `volumes: ["a"]` or `sysctls: ["a"]` —
both already reached by the existing `list-of-str` shape, and both already
appeared as over-rejects in `2026-07-15.02`'s first run.

## Non-goals

- Not fixing any of the underlying limitations (quoted booleans, compound
  durations, long-form `volumes`) — those stay deferred, per the decision.
  This only makes the harness's own reporting match what it already
  promises.
- Not replicating the original 30-cell manual audit's every string spelling
  in the generated matrix — one spelling per boolean key is sufficient
  signal, argued above.

## Testing

`just test-conformance`: **724 probes** (715 matrix + 9 corpus), **0 hard
violations**, **11 over-rejections**, all printed by the new terminal
summary:


All eleven are already catalogued in `planning/deferred.md`. `just test-ci`
still passes at 100% line coverage with the conformance suite deselected and
omitted from coverage. `just lint-ci` clean.

## Risk

- **The single `"true"` shape under-counts the boolean-key limitation's
  cells** relative to the original 30-cell manual audit. Accepted: the
  harness's job is to keep the limitation *visible*, not to re-run the
  original audit; one cell per key already does that, and re-measuring
  `"yes"`/`"on"` against Docker directly (done above, once, by hand) is
  cheaper than carrying redundant probes in CI forever.
