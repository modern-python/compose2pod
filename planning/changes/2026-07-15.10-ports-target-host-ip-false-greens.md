# Change: Close two false greens in `ports` long-form validation

**Lane:** lightweight ‚Äî one file (`compose2pod/values.py`), two field
validators, two new tests plus two corpus fixtures.

## Goal

Task 10 (`2026-07-15.09`) added value grammars for every long-form `ports`
field. A review of that diff found two of the new grammars still accept a
document `docker compose config` v5.1.2 refuses ‚Äî the exact false green
`planning/decisions/2026-07-14-docker-rejection-parity.md` forbids.

## Approach

Both measured against live `docker compose config` v5.1.2, same YAML text to
both oracles (`compose2pod.cli._read_compose` ‚Üí `compose2pod.parsing.validate`):

- **`target`'s string branch** used Python's `int()`, which strips
  surrounding whitespace and accepts digit-grouping underscores. Go's
  `strconv.Atoi` (what Docker actually runs, at interpolation time) accepts
  neither: `target: "1_000"` and `target: "  80  "` are both **REJECT**ed by
  Docker, but were **ACCEPT**ed here. Fix: reuse the existing
  `_STRICT_INT_STRING` regex (`^[+-]?[0-9]+$`, already used for `mode`'s
  `strict_string` path) instead of `int()`. No upper-bound change ‚Äî Task 10
  already measured (and this change re-confirms) that long-form `target` has
  no upper bound, unlike the short-form container port's 1-65535.

- **`host_ip`** used `ipaddress.ip_address`, which ‚Äî since Python 3.9 ‚Äî
  parses an RFC 4007 zone-id suffix (`fe80::1%eth0`, scope_id syntax).
  Docker's Go `net.ParseIP` has no zone-id support and refuses it (`invalid
  ip address`). Fix: reject any value containing `%` before it reaches
  `ipaddress.ip_address` ‚Äî `%` cannot appear in any v4/v6 literal Docker
  itself accepts, so this cannot over-reject.

Boundary re-measured beyond the two asked-for cases, to confirm the `%`
guard doesn't over-reject a form Docker accepts: a bracketed `[::1]` and a
whitespace-padded `"  1.2.3.4  "` were already rejected pre-fix (unchanged);
an IPv4-mapped `::ffff:1.2.3.4` (no `%`) is accepted by both oracles and
stays accepted.

## Files

- `compose2pod/values.py` ‚Äî `_validate_port_target_field`'s string branch
  switched from `int()` to `_STRICT_INT_STRING`; `_validate_port_host_ip_field`
  gained a `"%" not in value` guard ahead of the `ipaddress.ip_address` parse.
- `tests/test_values.py` ‚Äî `test_validate_ports_long_form_target_rejects`
  parametrized with `"1_000"`/`"  80  "`;
  `test_validate_ports_long_form_host_ip_rejects` parametrized with
  `"fe80::1%eth0"`/`"  1.2.3.4  "`; `test_validate_ports_long_form_host_ip_accepts`
  parametrized with `"::ffff:1.2.3.4"` (regression guard for the boundary
  re-measured above).
- `tests/conformance/corpus/ports_target_underscore_string.yaml`,
  `tests/conformance/corpus/ports_host_ip_zone_id.yaml` ‚Äî new corpus
  documents guarding both classes end to end.
- `architecture/supported-subset.md` ‚Äî `target`/`host_ip` prose updated to
  state the strict-string and zone-id-rejection grammars.
