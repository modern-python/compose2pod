# Design: True up the docs against the shipped docker-rejection-parity behavior

## Summary

`planning/audits/2026-07-14-docker-rejection-parity.md` found three places
where the tree contradicts the code (D1-D3); a differential re-audit during
Task 4 found two more (a stale `_sweep_document` docstring claim, and a gap
in test coverage that a later task already closed). This change fixes the
three D-findings, confirms the coverage gap was already closed, fixes the
stale docstring, and promotes `architecture/supported-subset.md` to
describe the behavior the `docker-rejection-parity` decision's five
implementation tasks actually shipped: ignored-service-key shape checks,
per-key resource-limit grammars, and the structural-key tightening. Docs
and comments only -- no runtime behavior changes.

## Motivation

`architecture/README.md` requires the capability page to describe the
system's *current* behavior, edited in the same PR as the code that changes
it. Five tasks landed real behavior changes on this branch
(`5480aff`..`0735814`) but the architecture page and one gate docstring
still asserted claims the code no longer matches:

- **D1:** `supported-subset.md` said, in one place, that a null on
  `entrypoint`/`command`/`deploy` is accepted (correct,
  `NULL_ALLOWED_KEYS`), and in another that `entrypoint: null` raises and
  that `command: null` acceptance is a divergence "unlike entrypoint" --
  both false; `_validate_entrypoint`/`_validate_command` treat a null on
  either identically, as "not specified".
- **D2:** `_require_string_keys_deep`'s docstring (`parsing.py`) claimed
  rejecting a non-string map key is "a deliberate divergence from Docker...
  (Docker accepts `environment: {3306: db}`)". Measured against `docker
  compose config` v5.1.2: it does not -- `non-string key in
  services.app.environment: 3306`. `2026-07-14.09` corrected the page but
  not this docstring, so the two disagreed.
- **D3:** the page named `extra_host_pairs`; the function was renamed to
  `keys.extra_host_entries` in `2026-07-14.06`.
- A same-class bug found while fixing D2/D1: `_sweep_document`'s docstring
  said the top-level `networks`/`volumes` blocks are "never read", but
  Task 4 added `_validate_network_references`, which reads the top-level
  `networks` block's keys to check per-service references. The identical
  claim was duplicated in the architecture page's own prose.
- The resource-limits table still showed a blanket "number or string"
  shape for all thirteen legacy keys, even though Tasks 1-3 gave each one
  a distinct, Docker-measured grammar (size / number / count / integer /
  string). The table no longer matched `compose2pod/values.py`.
- No section stated the governing rule
  (`planning/decisions/2026-07-14-docker-rejection-parity.md`) or pointed
  at `tests/conformance/`, even though both now exist and are what the
  rest of the page's parity claims rest on.

## Design

**D1.** Rewrote the `command`/`entrypoint` structural-key bullets in
`supported-subset.md` to state a null is accepted and treated as absent for
both, matching Docker and `NULL_ALLOWED_KEYS`. Left the Extends section's
null-as-reset language untouched -- it was already correct.

**D2.** Rewrote `_require_string_keys_deep`'s closing paragraph
(`compose2pod/parsing.py`) to state the rule matches Docker (with the
measured error string as evidence), and kept the still-true reason the
YAML-1.2 boolean resolver exists: a bare `on:` would otherwise resolve to
the bool `True` under PyYAML's default YAML-1.1 resolver and be refused --
rejecting a file Docker (parsed as YAML 1.2) runs.

**D3.** `extra_host_pairs` -> `extra_host_entries` in the one place the
stale name survived (the corruption-not-crash example list).

**Stale `_sweep_document` claim.** Reworded both the docstring
(`parsing.py`) and the matching prose in `supported-subset.md` so neither
claims the top-level `networks` block is unread. Both now say its contents
stay unread, but `_validate_network_references` reads its *keys* for a
membership test -- which needs no string-key sweep, since a membership test
neither sorts, `startswith`s, nor f-string-interpolates the key the way the
sweep's other protected readers do.

**Missing `_validate_build` test.** Checked first, per the task brief:
`tests/test_parsing.py::test_build_mapping_rejects_non_string_key` already
covers `build: {True: "x", context: "."}` raising `UnsupportedComposeError`
cleanly. Nothing to add.

**The promotion.** In `supported-subset.md`:

- Added a "Docker-rejection parity" section (new, placed right after the
  intro) stating the governing rule in both directions, the document-only
  binding (with the `env_file`/`${VAR:?msg}` host-dependent carve-out as
  the example), and pointing at `tests/conformance/` for enforcement.
- Rewrote the "Ignored (warns)" service-key bullet to state that all seven
  ignored keys are shape-checked, not just warned about, and to name each
  key's grammar (string / boolean / duration / list-of-strings / the port
  grammar) -- including that `restart`/`stop_signal` stop at a type check
  because that's all Docker itself validates for either (measured).
- Replaced the resource-limits table's blanket "number or string" column
  with each key's actual grammar (size / number / count / integer /
  string), added prose defining each grammar and citing its `values.py`
  function, and corrected the `deploy.resources` bullet, which previously
  claimed the same shape as the (now-tightened) legacy keys -- it still
  uses the older, looser `is_number`-based check, a real and previously
  undocumented asymmetry between the two paths to the same podman flags.
- Verified (not changed) that the `image`/`container_name` empty-string
  rejection, the per-service-network-must-be-declared-top-level check, the
  `dns_opt`-requires-a-list asymmetry, and the YAML-1.2 float resolver
  section were already accurate and current -- Tasks 1-4 had already
  landed them into the page. Left as-is per the page's own convention:
  tighten, don't re-say.

## Non-goals

Any runtime behavior change. `deploy.resources`'s looser grammar is
documented as-is, not tightened -- closing that gap (if it is one; the
audit's matrix does not probe nested `deploy.resources.*` positions at
all) is separate work, outside this change's scope.

## Testing

`just test-ci` (100% coverage, unchanged pass count -- no test added, since
M2 was already covered) and `just lint-ci` (ruff `--no-fix`, ruff format,
ty, eof-fixer, planning check), both clean. Re-read
`architecture/supported-subset.md` end to end after editing to confirm no
sentence contradicts `compose2pod/parsing.py`, `values.py`, `keys.py`, or
`resources.py` -- the same self-contradiction check D1 exists to catch.

## Risk

Low. Doc and comment text only; no `compose2pod/*.py` runtime logic
changed (the two `parsing.py` edits are both docstrings). The main risk was
overclaiming a grammar in the resource-limits table that doesn't match
`values.py`'s actual validators -- mitigated by reading each validator
(`validate_size`/`validate_number`/`validate_count`/`validate_integer`/
`validate_string`) and its registry wiring (`keys.py`'s `SERVICE_KEYS`
entries) directly before writing the table, rather than trusting the
audit's prose summary.
