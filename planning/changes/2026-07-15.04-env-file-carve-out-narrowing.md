# Design: Narrow the `env_file` carve-out to its host-dependent shapes

## Summary

`tests/conformance/test_matrix.py` excluded the entire `env_file` key from
the probe matrix via `CARVE_OUT_KEYS = {"env_file"}`, on the grounds that
`docker compose config` rejects a missing env file as a fact about the
*reading host's* filesystem, not the document
(`planning/decisions/2026-07-14-docker-rejection-parity.md`). That carve-out
is correct for the shapes that put a bare path string in front of Docker's
file-existence check, but excluding the whole key also hid a genuine,
host-independent over-rejection: compose2pod does not parse `env_file`'s
long-form mapping (`{path, required, format}`), which Docker accepts
unconditionally when `required: false`. The carve-out is now scoped to
exactly the shapes that are host-dependent; every other `env_file` shape is
probed like any other key.

## Motivation

A key-wide carve-out cannot distinguish "Docker's rejection here is about the
host" from "Docker's rejection here is about the document" — it silently
swallows both. `planning/deferred.md` already documents this exact class of
gap for `volumes` (long-form mapping, list-of-str) and quoted booleans; a
matching `env_file` gap sat unreported because the carve-out excluded the key
outright rather than only the shapes that actually needed it.

## Design

**Which shapes are host-dependent, verified by hand against
`docker compose config` v5.1.2:**

- `env_file: ""` (empty-str), `env_file: "somevalue"` (str),
  `env_file: "true"` (quoted-bool), `env_file: ["a"]` (list-of-str) — all four
  raise `env file ... not found: stat ...: no such file or directory`. Every
  one puts a string in front of Docker's existence check, regardless of the
  string's content (even the empty string reaches the check).
- `env_file: []` (empty-list) does **not** — zero entries means zero paths to
  check, so it is not host-dependent and needed no carve-out (confirmed: both
  oracles already agreed on this shape before and after the change).

`test_matrix.py`'s `CARVE_OUT_KEYS = {"env_file"}` (a `set[str]`, filtered at
the `key` parametrize) became `CARVE_OUT: set[tuple[str, str]]` holding the
four `(key, shape)` pairs above, checked inside the test body via
`pytest.skip` (mirroring the existing `_require_docker` autouse-skip idiom in
`conftest.py`) rather than filtering at collection time — collection-time
filtering only composed over one axis (`key` alone), and skipping specific
`(key, shape)` pairs needs both.

**The gap this surfaced:** `tests/conformance/corpus/env_file_long_form.yaml`
— `env_file: [{path: app.env, required: false}]`. Docker accepts this
unconditionally (`required: false` means the existence check plays no part;
verified: exit 0 whether or not `app.env` exists). compose2pod's
`_validate_string_or_string_list` (`compose2pod/parsing.py`) requires every
`env_file` entry to be a `str`, so it refuses with `'env_file' entry must be
a string`. Added to the corpus (not the generated matrix) for the same reason
`volumes_long_form.yaml` is: the matrix's generic shapes
(`list-of-map: [{"a": 1}]`) cannot stand in for a domain-specific field name
(`path`) — verified by hand that `[{"a": 1}]` is `both-reject` (Docker:
`"environment path attribute ... is missing"`), not `over-reject`.
`required: false` keeps the corpus document itself host-independent, so no
change was needed to `conftest.py`'s file-writing (it only ever materializes
`compose.yaml`, never a referenced env file).

**Catalogued:** `planning/deferred.md` gains a "Long-form `env_file`" bullet
alongside "Long-form `volumes`", noting the mapping resolves to the same
`--env-file <path>` flag `emit._env_flags` already emits for the string form,
so `required`/`format` are the only real parsing work.

## Non-goals

Implementing long-form `env_file` support. This change only makes the
limitation visible in the harness and catalogues it; closing it is separate
work.

## Testing

`just test-conformance`: 738 selected (up from 724) — `env_file` re-enters
the `key` parametrize (56 keys × 13 shapes = 728 matrix combinations, 4 of
them skipped via `CARVE_OUT`) plus the new `env_file_long_form` corpus file
(10 corpus documents total): 728 + 10 = 738. Result: 734 passed, 4 skipped, 0
failed — 0 hard violations. 12 over-rejections (the prior 11, unchanged,
plus `env_file_long_form`) — every one cross-checked against
`planning/deferred.md` and found catalogued.

## Risk

Low — the change only widens what the matrix probes and adds one corpus
fixture; it does not touch `compose2pod`'s runtime code. The main risk was
under-scoping the carve-out (missed exactly this on the first pass: only
`str`/`list-of-str` were excluded initially, and `empty-str`/`quoted-bool`
surfaced as false hard violations before the fix above) — mitigated by
running the full matrix and confirming 0 hard violations before considering
the change done.
