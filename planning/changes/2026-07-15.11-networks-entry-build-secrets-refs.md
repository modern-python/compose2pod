# Design: networks entry sub-schema and build.secrets source cross-reference

## Summary

Two remaining nested false-green classes, both flagged as known gaps by
earlier changes (`2026-07-15.09`'s non-goals, `_validate_build_secrets`'s own
docstring): a per-service long-form `networks` entry (`networks: {n: {...}}`)
had no sub-key schema at all beyond `aliases`, and `build.secrets` never
cross-checked its `source` against the top-level `secrets:` block. Both are
document-wide checks Docker enforces regardless of whether compose2pod reads
the value; both close with the established patterns — a strict-key-schema
dict (Task 9's `_DOCKER_BUILD_KEYS`) for the first, a document-wide
cross-reference function (Task 4's `_validate_network_references`) for the
second.

## Motivation

- **`networks` entry sub-schema.** `graph._host_names` only ever read
  `aliases` from a long-form network entry; a non-mapping value
  (`networks: {n: somevalue}`) or an unknown sub-key (`networks: {n:
  {badkey: 1}}`) sailed through unchecked. Measured against `docker compose
  config` v5.1.2, both are refused ("must be a mapping" / "additional
  properties 'badkey' not allowed" — a STRICT schema, unlike short-form
  `networks`).
- **`build.secrets` source reference.** `_validate_build_secrets`
  (`2026-07-15.08`) already validated each entry's shape (a string, or a
  `{source, target}` mapping), but its own docstring flagged the cross-check
  as unreproduced: "compose2pod never reads `build`'s contents at emit
  time... closing that gap would need `_validate_build` to see the whole
  document." Measured: `build: {secrets: [nope]}` with no top-level `secrets:
  {nope: ...}` is refused ("refers to undefined build secret nope"), for
  both the short (bare string) and long (`{source, target}`) forms.

## Design

**`networks` entry sub-schema** (`compose2pod/parsing.py`): the full valid
key set was measured by probing individual candidate keys against live
`docker compose config` v5.1.2, then cross-checked against the upstream
compose-spec JSON schema (`$defs/service.networks.oneOf[1].patternProperties`)
via its raw GitHub source — both agree on exactly 9 keys, `additionalProperties:
false`, plus a `^x-` extension pattern: `aliases` (list of strings),
`ipv4_address`/`ipv6_address`/`mac_address`/`interface_name` (plain string,
content unchecked), `priority`/`gw_priority` (native number, no string
form), `link_local_ips` (list of strings), `driver_opts` (mapping, each
value a number or string). This is one more key than the task brief's
initial list — `interface_name` was found by probing beyond the brief's
enumeration, exactly the failure mode "measure the complete set yourself"
exists to catch.

New `_DOCKER_NETWORK_ENTRY_KEYS` dict (mirrors `_DOCKER_BUILD_KEYS`'
strict-key-schema pattern) wires each key to a validator: `_validate_string_list`
(shared with `link_local_ips`), `values.validate_string`, or two new
functions — `values.validate_native_number` (`priority`/`gw_priority`) and
`_validate_network_driver_opts`. New `_validate_network_entry_value` checks
one entry (`None` accepted — measured, means "use default settings", same
as `{}`; non-mapping raises; unknown non-`x-` key raises; each known key's
value runs through its validator), and `_validate_network_entries` sweeps
every service's long-form entries document-wide — mirroring
`_validate_network_references`'s own document-wide scope (not the
startup-order closure `emit.py` computes at emit time), called from
`validate()` right after it. `graph._host_names`' own `aliases` check is
untouched — the two now enforce the same grammar twice, the same
defense-in-depth pattern Task 10 established for `extra_hosts`.

`values.validate_native_number` is new and deliberately does **not** apply
this module's usual `has_variable` carve-out. Measured: `priority: "${VAR}"`
is refused by Docker even with the variable set to a valid number in the
reading shell (interpolation always produces a string, and this field has no
string branch at all) — so unlike every other grammar in `values.py`, no
possible variable value could make Docker accept the document, and the
rejection is a fact about the document alone, not the host. Applying the
usual carve-out here would itself be a false green.

**`build.secrets` reference** (`compose2pod/parsing.py`): new
`_validate_build_secret_references(compose, services)`, run in `validate()`
after `stores.validate(compose)` (so the top-level `secrets:` block, if
present, is already confirmed to be a mapping) and after the per-service
loop (so each `build.secrets` entry's own shape is already confirmed by
`_validate_build_secret_entry`). For each service's `build.secrets` entries,
extracts the source (the bare string itself, or the mapping's `source`
field) and checks membership in the declared top-level `secrets:` names,
raising "refers to undefined secret" otherwise — the same verdict shape as
`_validate_network_references`.

## Non-goals

Reusing `stores.py`'s `_ref_source`/`_LONG_FORM_KEYS` machinery wholesale for
`build.secrets` was considered and rejected: that machinery accepts
`uid`/`gid`/`mode` fields real service/config references have but
`build.secrets` entries measurably do not (`_BUILD_SECRET_REF_KEYS =
{"source", "target"}`, unchanged, already enforced) — reusing it would
silently widen build.secrets' accepted shape. Only the tiny
string-or-mapping source-extraction idiom is shared (already duplicated
inline three times in `stores.py` itself).

## Testing

**Unit**: `tests/test_values.py` (`validate_native_number` — accepts,
rejects including numeric strings and bools, and the deliberate
non-carve-out for `${VAR}`), `tests/test_parsing.py`
(`TestNetworkEntrySchema` — non-mapping, unknown key, null/empty accepted,
extension key, aliases defense-in-depth, all four string fields,
priority/gw_priority native-number-only, link_local_ips, driver_opts,
the `${VAR}`-can-never-produce-a-mapping case, and a crash-class guard for a
list value at the entry position; four `build_secrets_*` tests for the
undeclared-source reject on both forms, the declared-source accept on both
forms, and the no-top-level-block case). Confirmed every reject-side test
failed (`DID NOT RAISE`) against pre-fix code, then passed after the fix.
`just test-ci`: 1172 passed, 100% line coverage.

**Conformance** (`tests/conformance/corpus/`): `network_entry_non_mapping`,
`network_entry_unknown_key`, `network_priority_numeric_string`,
`network_driver_opts_bool_value`, `build_secrets_undefined_source` (short
form), `build_secrets_undefined_source_long_form`. All 6 confirmed failing
(`AssertionError: docker compose config REJECTS this document but
compose2pod ACCEPTS it`) against pre-fix code via `git stash`, then passing.
`just test-conformance`: 776 selected, 772 passed, 4 skipped, **0 hard
violations**. 12 over-rejections, all pre-existing (quoted booleans,
`sysctls`/`volumes` list-of-string edges, `env_file`/duration host-state
carve-outs) — identical to `2026-07-15.10`'s list, none newly introduced.

`just lint-ci` and `just check-planning` both clean.

This closes the last nested region flagged by the sweep
(`2026-07-15.09`'s non-goals): the whole-document hard rule
(`accepted(compose2pod) ⊆ accepted(docker)`) is now believed closed.

## Risk

Low — both grammars only narrow what was previously accepted unconditionally,
the direction the parity rule requires. The main risk was assuming the
brief's key list was complete without probing further, which is why
`interface_name` was checked for and found; the JSON-schema cross-check adds
independent confirmation the measured set has no more gaps.
