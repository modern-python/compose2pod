# Design: Close the healthcheck scalar grammar gate

## Summary

`_validate_service_healthcheck` (`compose2pod/parsing.py`) gates `timeout`,
`retries`, `start_period` through `keys.is_number` (`isinstance(value, int |
float | str)` — any string passes) and gates `interval` through
`healthcheck.interval_seconds`, whose bare-number fallback silently treats
any unitless numeric string, and any native number, as a count of seconds.
Both are false greens: `docker compose config` v5.1.2 refuses a native
number and a unitless string on all four keys. This routes the three scalar
keys through `values.validate_duration` / `values.validate_count` (already
written, tested grammars used elsewhere in the file) and rewrites
`interval_seconds` to enforce the same mandatory-unit rule directly.

## Motivation

`planning/decisions/2026-07-14-docker-rejection-parity.md` binds
`accepted(compose2pod) ⊆ accepted(docker)` with no exceptions. A whole-branch
review after that decision's rollout found the flat (top-level) gate closed
but these four *nested* healthcheck positions still on the old loose checks
— `interval: 30`, `timeout: somevalue`, `start_period: 30`, `retries:
somevalue` were all silently accepted, each a document `docker compose`
itself refuses. Measured against a live `docker compose config` v5.1.2
oracle (both oracles reading the same YAML text, `compose2pod.cli.
_read_compose` then `compose2pod.parsing.validate`):

| key | value | Docker |
|---|---|---|
| `timeout`, `start_period` | `30s`, `1h30m`, `1h`, `500ms`, `'0s'`, `'0'` | ACCEPT |
| `timeout`, `start_period` | `30`, `'30'`, `1.5`, `'1.5'`, `0`, `3`, `'3'`, `somevalue` | REJECT |
| `interval` | `30s`, `500ms`, `'0'` | ACCEPT |
| `interval` | `30`, `'30'`, `1.5`, `0`, `3`, `'3'`, `0.5`, `somevalue` | REJECT |
| `retries` | `3`, `'3'`, `30`, `'30'`, `1.5`, `0.5`, `0`, `'0'` | ACCEPT |
| `retries` | `'1.5'`, `30s`, `1h30m`, `somevalue` | REJECT |

`values.validate_duration` additionally over-rejected `'0'` (Go's
`time.ParseDuration` special-cases the zero literal; Docker accepts it,
measured), which surfaced identically on `stop_grace_period: '0'`.

## Design

**`timeout`/`start_period`.** `_HEALTHCHECK_SCALAR_KEYS` now dispatches
through a small `key -> validator` map instead of the uniform `is_number`
check:

```python
_HEALTHCHECK_SCALAR_VALIDATORS: dict[str, Callable[[str, str, Any], None]] = {
    "timeout": values.validate_duration,
    "start_period": values.validate_duration,
    "retries": values.validate_count,
}
```

`values.validate_duration` is the same Go-duration grammar already used for
`stop_grace_period` — mandatory unit, compound and hour forms both accepted,
matching the measured table exactly with no extra restriction layer.

**`retries`.** Routed through `values.validate_count`, already written for
`cpu_shares`/`cpu_quota`/`cpu_period`/`pids_limit`: a native number casts
leniently (`0.5` accepted), a string form must be `strconv.ParseInt`-strict
(no decimal point, no exponent). This is an exact match for the measured
`retries` row with zero new code.

**`interval`.** Stays on `healthcheck.interval_seconds` (it must still
compute a polling-loop integer, and deliberately reject the compound/hour
forms `timeout`/`start_period` accept — a separate, already-deferred
limitation, `planning/deferred.md`). The function's native-number branch
(`isinstance(duration, (int, float))`) is deleted; any non-`None`,
non-`str` value now raises outright, and the string path requires an
explicit `ms`/`m`/`s` suffix — except the literal `"0"`, mirrored from the
`validate_duration` fix below. This closes the false green without touching
the deliberate compound/hour rejection, which was already suffix-based and
unaffected.

**`values.validate_duration` `'0'` fix.** `_DURATION.match(value)` mandates
a unit, so `'0'` (no unit) was rejected — but Docker accepts it as Go's
zero-duration special case. Fixed by special-casing the exact string `"0"`
ahead of the regex match; `'30'` (or any other unitless string) still fails
the regex and is still refused. Verified `stop_grace_period` post-fix: `'0s'`
/ `'1m30s'` still accept, `'30'` / a native number still reject — same
`test_validate_duration_accepts`/`_rejects` suite, now covering `'0'` too.

**`keys.is_number`.** Left in place — `keys.py:122` (label-value validation)
still uses it; only its `parsing.py` import/usage is removed.

## Non-goals

- `start_interval` — an unsupported healthcheck key entirely; refusing it is
  the honest subset, not a false green. Untouched.
- The `interval: '1h30m'`/`'1h'` over-rejection — Docker accepts both;
  compose2pod's refusal is `planning/deferred.md`'s existing, deliberate
  deferral. Not revisited here.
- `healthcheck.test` (`health_cmd`) — already correct, not part of this gate.

## Testing

- `tests/test_values.py`: `'0'` added to `test_validate_duration_accepts`;
  `test_validate_duration_accepts_bare_zero_string` and
  `test_validate_duration_rejects_native_number` pin the fix's boundary
  (`'0'` in, native `0`/`30` and string `'30'` still out).
- `tests/test_healthcheck.py`: `test_native_number_raises` and
  `test_unitless_string_raises` replace the old
  `test_int_value_passes_through`/`test_float_value_below_one_floors_to_one`
  (which asserted the bug); `test_bare_zero_string_floors_to_one` pins `'0'`.
- `tests/test_parsing.py`: per-key accept/reject parametrized tests for all
  three measured rows above, routed through `validate()`; the three existing
  `*_rejected_at_gate` tests updated for the new (more precise) error
  messages.
- `tests/conformance/corpus/`: four new files —
  `healthcheck_interval_bare_number.yaml`,
  `healthcheck_timeout_non_duration_string.yaml`,
  `healthcheck_start_period_bare_number.yaml`,
  `healthcheck_retries_non_numeric_string.yaml` — each a false green the old
  code silently accepted; `just test-conformance` now reports them
  `both-reject`, 0 hard violations (745 passed, 4 skipped, 12
  over-rejections, all pre-existing and catalogued in `planning/deferred.md`).
- `just test-ci`: 100% line coverage. `just lint-ci`: clean.

## Risk

Low — every branch is measured against a live `docker compose config`
v5.1.2 oracle before being written into a grammar (see the table above), and
the two functions this touches (`validate_duration`, `validate_count`) are
pre-existing, already-tested code, not new logic. The one genuine risk was
regressing `stop_grace_period` while fixing the shared `validate_duration`
function; mitigated by re-running its existing accept/reject suite with the
fix in place before considering the change done.
