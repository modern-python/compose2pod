# Design: Four nested false-green classes

## Summary

A comprehensive sweep for `planning/decisions/2026-07-14-docker-rejection-
parity.md`'s hard rule (`accepted(compose2pod) ⊆ accepted(docker)`, no
exceptions) found four remaining classes where a mis-typed nested value was
accepted here and refused by `docker compose config`: `build.ssh`'s list
form, a non-string `extra_hosts` map value, secret/config reference
`mode`/`uid`/`gid`, and `ports` long-form entry fields. Each closes with its
own grammar, following the pattern `2026-07-15.06`–`.08` already established
for deploy/healthcheck/build.

## Motivation

- **`build.ssh`** was wired to the exact same list-or-map validator as
  `args`/`labels` (`2026-07-15.08`), on the assumption all three share one
  grammar. Measured: they don't. `build.ssh: [mykey]` is refused (`invalid
  ssh key "mykey"`) — a bare list entry must equal `default`; `args`/`labels`
  accept any bare key as a null value. This is a regression the previous
  change introduced, not a pre-existing gap.
- **`extra_hosts`** (per-service, `compose2pod/pod.py`) runs a map value
  through `keys.validate_map`'s shared scalar-or-null shape check — correct
  for `labels`/`annotations`' bare-key-means-null semantics, wrong here:
  Docker refuses `extra_hosts: {h: 3}` and `extra_hosts: {h: true}`
  ("must be a string"), and the gate never enforced it.
  `keys.extra_host_entries`'s `_render_scalar` then silently coerced the
  int/bool into the emitted `--add-host` value.
- **Store references** (`compose2pod/stores.py`) treated `mode`/`uid`/`gid`
  as one grammar: "an int or string, not a bool". Measured: they are two
  different grammars. `mode` goes through Go's `strconv.ParseInt` at decode
  time (native int or a strict ParseInt-grammar string; a non-numeric string
  like `mode: somevalue` fails). `uid`/`gid` are a plain string field with
  *no* further parsing at config-validate time — a native int is refused
  outright (`must be a string`), while the string's content is entirely
  unchecked, even `uid: somevalue`.
- **`ports` long-form entries** (`values._validate_port_entry`) checked only
  that `target` was present, treating every other field as "Docker's
  business, not ours". Measured: `protocol: 3` and `host_ip: 3` are both
  refused (`must be a string`), and `host_ip` additionally must parse as a
  real IP address (`invalid ip address`).

## Design

**`build.ssh`** (`compose2pod/parsing.py`): new `_validate_build_ssh` calls
the existing `_validate_build_map` first (shape + string-key check, shared
with `args`/`labels`), then narrows further — for the list form only, a bare
entry (no `=`) must equal `'default'`; an entry with `=` accepts any id, and
the map form stays unrestricted. Wired into `_DOCKER_BUILD_KEYS["ssh"]` in
place of the shared validator.

**`extra_hosts`** (`compose2pod/pod.py`): new `_check_extra_host_value_types`
runs alongside the existing `_check_extra_host_separators`, after
`validate_map`, in `validate_pod_options`. Map form only (the list form is
already all-string, guaranteed by `validate_map`'s own list-element check);
rejects any value that isn't `isinstance(..., str)`, which naturally admits
`${VAR}` (itself a string) without a separate carve-out. `keys.
extra_host_entries`'s comment is corrected — it no longer claims Docker
"normalizes" a boolean address; it refuses one, and the function's own
`_render_scalar` call is now pure defense-in-depth for a caller that
bypasses the gate.

**Store `mode`/`uid`/`gid`** (`compose2pod/stores.py`): `_check_long_form_
scalars` now calls `values.validate_string` for `uid`/`gid` and `values.
validate_integer(..., strict_string=True)` for `mode`. `validate_integer`
gains a `strict_string` keyword: when set, the string branch matches against
the existing `_STRICT_INT_STRING` regex (already used by `validate_count`)
instead of Python's lenient `int()` cast — closing a whitespace gap
(`int("  400  ")` succeeds in Python, `strconv.ParseInt` does not).
`validate_string` needed no change; it already accepts any string with no
content check, which is exactly `uid`/`gid`'s measured grammar. Applies
identically to both `secrets` and `configs` references (measured).

**`ports` long form** (`compose2pod/values.py`): the dict branch of
`_validate_port_entry` is extracted into `_validate_port_long_form` (keeping
cyclomatic complexity down) and now validates every known field:
`target` (already required) — a non-negative integer, native or a
whole-valued float, or a string that parses as one, no upper bound (measured:
`target: 99999` passes; Docker casts a string through Go's uint32 decoder,
which is why a non-numeric string like `"abc"` fails even without a `${VAR}`
involved); `published` — an integer or a string with content entirely
unchecked (Docker's own field is a bare string; `published: abc` and
`published: "8080-8090"` both pass, only `published: true`/`80.5` fail the
int-or-string type union); `host_ip` — a bare IP address string, checked with
stdlib `ipaddress.ip_address` (accepts the same unbracketed v4/v6 forms
Docker does, rejects the same malformed ones: a hostname, `[::1]` bracketed,
a truncated dotted-quad); `protocol`/`mode`/`name`/`app_protocol` — a plain
string, content unchecked (measured: `mode: bogus` passes). An unrecognized
long-form key now raises too — Docker's own schema is strict here
(`additional properties '<key>' not allowed`), and `app_protocol` (a real
Compose Spec field, easy to mistake for an unknown one) is in the known set.

## Non-goals

Per the task brief: networks long-form value validation and `build.secrets`'
source cross-reference are separate, tracked gaps, untouched here.

## Testing

**Unit**: `tests/test_parsing.py` (ssh list-vs-map split, replacing the
now-wrong shared assertion `ssh: ["KEY"]` accepted), `tests/test_pod.py`
(extra_hosts int/bool/null/`${VAR}` map values), `tests/test_stores.py`
(uid/gid string-only + content-unchecked, mode strict-ParseInt, both
secrets and configs), `tests/test_values.py` (every long-form ports field,
accept and reject sides, plus the unknown-field and no-upper-bound cases).
Confirmed each fix's test failed against the pre-fix code, then passed.
`just test-ci`: 1130 passed, 100% line coverage.

**Conformance** (`tests/conformance/corpus/`): `build_ssh_bare_non_default`,
`extra_hosts_map_int_value`, `secret_mode_non_numeric_string`,
`secret_uid_native_int`, `ports_long_form_protocol_int`,
`ports_long_form_host_ip_invalid`. `just test-conformance`: 768 selected, 764
passed, 4 skipped, 0 failed — 0 hard violations; 12 over-rejections, all
pre-existing (quoted booleans, `sysctls`/`volumes` list-of-string edges,
`env_file`/duration host-state carve-outs), none touching this change.

`just lint-ci` and `just check-planning` both clean.

## Risk

Low — each of the four grammars only narrows what was previously accepted
unconditionally, and the parity rule forbids exactly the opposite direction.
The main risk, as with `2026-07-15.08`, is assuming a grammar from a
sibling's name rather than measuring it: `uid`/`gid` were assumed to share
`mode`'s "int-or-numeric-string" shape (the task brief's own working guess)
but measured out as a plain unchecked string instead — surfaced and matched
to the measured behavior rather than the guess.
