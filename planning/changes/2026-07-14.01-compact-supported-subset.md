# Design: Compact the supported-subset page

## Summary

`architecture/supported-subset.md` is the accept/ignore/reject truth for the
whole tool, and it is the page an agent or a contributor reads to answer "is
this key supported, and what does it do?". It has reached 765 lines, and a
large fraction of it is duplication and prose restating what a table would say
in a row. Rewrite it into the same content at roughly half the length: one
**Stores** section instead of near-identical Secrets and Configs twins, and a
table for the per-key flag mappings instead of one bullet each.

This is a documentation-only change. No code, no behavior.

## Motivation

Two measurements, taken on the page as it stands:

- **Secrets (59 lines) and Configs (66 lines) are twins.** The page says so
  itself — "mirroring secrets", "the same closure-scoped-creation rule secrets
  follow", "`uid`/`gid`/`mode` behave exactly as for secrets", "byte-for-byte
  the same teardown parity as secrets". They differ in exactly four ways: the
  store-name prefix, the allowed sources, the default mount target, and whether
  an absolute target is required. That is a four-row table, not 125 lines.
- **Service keys is 174 lines**, most of it bullets of the shape "`X` is a
  list, emitted as repeated `--x`" — `cap_add`, `cap_drop`, `security_opt`,
  `devices`, `group_add`, `platform`, `user`, `working_dir`, and the twelve
  numeric resource keys. Each row is: key, podman flag, accepted shape. That is
  a table.

The page grew 537 → 765 lines while closing the structural-key gate
(`2026-07-13.10`), because each new rule arrived as new prose. The content is
correct and worth keeping; the *form* is what costs.

Length is not a vanity metric here. This page is loaded to answer a lookup
("what shape does `ulimits` take?"), and a lookup is exactly what a table
serves and running prose does not.

## Design

Rewrite the page with the same section order, changing form not substance:

- **Stores (secrets and configs)** — one section. Open with the shared model
  (both render as podman secrets; podman has no config primitive), then a
  four-row table of the differences, then the shared rules stated once:
  closure-scoped creation, mounting, teardown, variable interpolation, and the
  reject list.
- **Service keys** — a table of the declarative registry keys (key → podman
  flag → accepted shape), and a table of the resource keys. Prose survives only
  for keys whose behavior a table cannot carry: `entrypoint`'s shell-vs-exec
  form and its interaction with `command` and the `--command` override;
  `hostname`/`container_name` resolving via `--add-host` rather than a
  per-container flag; the ignored-keys list and *why* each is inert.
- **Everything else** — keep as prose. The nuanced sections earn their length:
  Extends' merge policy, Pod-level options' pod-wide divergence, Variable
  interpolation, the mapping-key sweep and its scope, and the three deliberate
  Docker divergences (boolean values normalize; non-string keys reject;
  a null healthcheck scalar means unset).

Code citations drop from roughly one per bullet to one per section. They are
useful navigation but they are also the page's drift surface — a stale
`_MAP_FLAGS` citation (a symbol that no longer existed) is exactly what the
audit found.

## Non-goals

- **No rule may be lost.** This is a reformat, not a re-scope. Every shape the
  gate enforces and every divergence from Compose must still be stated. The
  review gate for this change is a mechanical cross-check against the code, not
  a reading of the old page.
- **No new rules, no code changes.** If the rewrite surfaces a rule the page
  never documented, note it — do not invent one, and do not change code to fit
  the page.
- Not touching `README.md`'s subset paragraph or `glossary.md` (separate audit
  findings, separate changes).

## Testing

`just check-planning` and `just lint-ci` (the eof-fixer and the planning check
cover this file). No test suite exercises prose.

The real gate is a **coverage cross-check**: enumerate every
`UnsupportedComposeError` message and every accepted shape in `compose2pod/`,
and confirm the rewritten page still states each one. A rule present in the
code and absent from the page is a defect in this change.

## Risk

- **Silent information loss** (the whole risk of this change): a rule that
  survives in code but vanishes from the page. Mitigated by the mechanical
  cross-check above rather than by careful reading, because careful reading is
  what let the page drift in the first place.
- Tables are worse than prose for anything conditional. Where a key's rule has
  a real "unless", it stays prose — resisting the urge to force every key into
  the table is part of the design, not a failure of it.
