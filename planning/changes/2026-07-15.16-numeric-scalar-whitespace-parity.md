# Change: numeric/size scalar whitespace parity

**Lane:** lightweight — one source file (`compose2pod/values.py`, three
grammars), plus a conformance-matrix shape, three corpus fixtures, and unit
tests. No new module, no public-API change.

## Goal

A PR #63 review found — and I re-measured against live `docker compose config`
v5.1.2 and confirmed — a false green of exactly the class
`planning/decisions/2026-07-14-docker-rejection-parity.md` forbids: a document
Docker rejects that compose2pod accepts, reachable through the public CLI.

Whitespace-padded numeric/size scalar strings were accepted where Docker
refuses them. Three grammars in `values.py` were more lenient about whitespace
than the Go parsers Docker uses.

## The boundary (measured, same YAML both oracles)

- **size** (`mem_limit`, `mem_reservation`, `shm_size`, ...): Docker's byte-size
  grammar allows **exactly one optional space, and only between the number and
  the unit**. No leading whitespace, no whitespace after the unit letter, no
  doubled space, no tab. Measured with a full lead×mid×unit×trail probe (69
  unique cases) against the live CLI:
  - `"512m"`, `"512 m"` — ACCEPT (one space before the unit is legal).
  - `" 512m"`, `"512m "`, `"512  m"`, `"512\tm"` — REJECT
    (`strconv.ParseFloat: ... invalid syntax`, or `invalid suffix` for the tab).
  - **One extra boundary cell, not in the review's table, found by the probe:**
    a bare number with a *single trailing space and no unit* (`"512 "`) ACCEPTS
    — because the unit group is optional, the lone space is the legal
    "before the (absent) unit" space, and there is nothing after it to make it
    trailing. `"512  "` (two spaces) and `"512\t"` (tab) still REJECT.
- **number** (`cpus`): Go's bare `strconv.ParseFloat` — **no surrounding
  whitespace at all.** `" 1.5 "`, `"1.5 "`, `" 1.5"` all REJECT; `"1.5"`
  ACCEPTs.
- **integer**, lenient branch (`oom_score_adj`): Go's `strconv.ParseInt` —
  **no surrounding whitespace at all.** `" 5 "`, `"5 "`, `" 5"` all REJECT;
  `"5"` ACCEPTs.

Python's `float()`/`int()` silently strip surrounding whitespace and the old
`_SIZE` regex anchored with `^\s*...\s*$`, so all the padded forms above were
over-accepted.

## Approach

Three fixes, all in `compose2pod/values.py`, each measured against Docker's
exact boundary rather than a blanket "strip and reparse":

1. **`_SIZE` regex.** Was
   `^\s* {_DIGITS}... \s* (?:[bkmgtp]b?)? \s* $`. Now
   `^ {_DIGITS}... \ ? (?:[bkmgtp]b?)? $` — plain `^`/`$` anchors (no
   surrounding whitespace) and a single optional *literal space* (` ?`, not
   `\s`, so no tab) in place of the pre-unit `\s*`. Reproduces all 69 measured
   size cells exactly, including the `"512 "` bare-number-trailing-space cell.

2. **`validate_number`.** Guarded the string branch with
   `value == value.strip()` before `float(value)`, since Go's `ParseFloat`
   permits no surrounding whitespace and Python's `float()` does.

3. **`validate_integer`, lenient (non-`strict_string`) branch only.** Added
   `value == value.strip()` alongside the existing `"_" not in value` guard
   before `int(value)`. The `strict_string=True` branch (`_STRICT_INT_STRING`,
   used by secret/config `mode`) was already anchored and correct — untouched.

`validate_count`, `validate_duration`, and `_STRICT_INT_STRING` were confirmed
already anchored and were not changed.

The `${VAR}` carve-out is unchanged: a value carrying a variable reference still
short-circuits ahead of every one of these checks (`mem_limit: "${MEM}"`
passes), since Docker's rejection of an unset-variable size is a fact about the
reading host, not the document.

## Regression guard: a whitespace matrix shape

The review noted the root cause the harness missed this: every existing
`SHAPES` entry in `tests/conformance/test_matrix.py` is structural
(null/empty/bool/int/list/map/bare-string) — none carried padded or internal
whitespace, so the generated matrix never probed the class. Added
`"padded-scalar": " 512m "`, which reaches every key's own grammar.

Adding it surfaced exactly one hit beyond the three target grammars:
`env_file`. Investigated (per the parity decision's process) rather than
silently fixed. `docker compose config` rejects `env_file: " 512m "` with
`env file ... not found: stat ... no such file or directory` — it reads the
padded scalar as a (missing) filesystem path. That is the *same* host-state
file-existence carve-out already applied to `env_file`'s other bare-string
shapes (`empty-str`, `str`, `quoted-bool`, `list-of-str`): a fact about the
reading host, not the document, which the decision explicitly does not let
bind. Added `("env_file", "padded-scalar")` to `CARVE_OUT` alongside the
existing four, with the maintainer's independent confirmation that it is
consistent precedent, not suppression.

## Files

- `compose2pod/values.py` — `_SIZE` regex re-anchored; `validate_number` and
  `validate_integer`'s lenient branch each guard surrounding whitespace before
  parsing. Docstrings updated to state the boundary.
- `tests/test_values.py` — `test_validate_size_accepts_single_space_before_unit`
  (`["512 m", "512 mb", "512 "]`, the ACCEPT side), and
  `test_validate_size_rejects_whitespace_padding` /
  `test_validate_number_rejects_whitespace_padding` /
  `test_validate_integer_rejects_whitespace_padding` (the REJECT rows). All
  reject-rows confirmed red pre-fix, green post-fix.
- `tests/conformance/test_matrix.py` — new `padded-scalar` shape;
  `("env_file", "padded-scalar")` added to `CARVE_OUT`.
- `tests/conformance/corpus/mem_limit_padded_size_string.yaml`,
  `cpus_padded_number_string.yaml`,
  `oom_score_adj_padded_integer_string.yaml` — new fixtures, one per grammar.
- `architecture/supported-subset.md` — the size/number/integer bullets in the
  Resource limits section now state the whitespace boundary precisely.

## Testing

**Unit** (TDD): the three `*_rejects_whitespace_padding` parametrizations (18
cells total) confirmed `DID NOT RAISE` against pre-fix `values.py`, all pass
post-fix; the ACCEPT-side guard passes both before and after (so the fix
cannot over-reject the legal one-space form). `${VAR}` carve-out reconfirmed
for all three functions post-fix.

**Conformance**: the three new corpus fixtures pass post-fix; the
`padded-scalar` matrix shape passes every non-carved key. `just
test-conformance` — 0 hard violations, 13 over-rejections (12 pre-existing,
carried forward, plus the pre-existing Windows drive-letter one; the
`padded-scalar-env_file` cell is now a carve-out skip, not a failure).

**Gates**: `just test-ci` — 100% line coverage. `just lint-ci` and `just
check-planning` clean.

## Risk

Low, and in the sound direction: this only *narrows* acceptance, closing a
false green. The one-space size form Docker accepts (`"512 m"`) and the
bare-number-trailing-space form (`"512 "`) both stay accepted, measured
explicitly so the fix does not overshoot into an over-rejection.
